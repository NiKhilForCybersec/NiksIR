Detection Principles

Good detection rules balance sensitivity (catching threats) with specificity (avoiding false positives). The goal is actionable alerts that analysts can investigate effectively.

Detection Goals

  • Detect real threats (true positives)
  • Minimize noise (false positives)
  • Provide actionable context
  • Enable efficient investigation
  • Cover attack techniques (MITRE)

Detection Challenges

  • Attackers blend with normal activity
  • Environment-specific baselines
  • Log availability and quality
  • Performance/cost constraints
  • Evolving attack techniques

Detection Quality Metrics

Metric Definition Target
True Positive Rate % of real threats detected As high as possible
False Positive Rate % of alerts that aren't threats < 20% ideal, < 50% acceptable
Alert Volume Alerts generated per day Manageable by team capacity
Mean Time to Triage Time to determine TP/FP Minutes, not hours

Alert Types

Signature-Based ‚ñº

Match specific known indicators or patterns.

Examples:

  • Known malicious IP address
  • Specific malware hash
  • Known attack tool command line

Pros:

Low false positives for known threats

Cons:

Misses new/modified attacks
Threshold-Based ‚ñº

Alert when count exceeds defined threshold.

Examples:

  • Failed logins > 10 in 5 minutes
  • File deletions > 100 in 1 hour
  • Outbound connections to new IPs > 50/day

Pros:

Catches volume-based attacks

Cons:

Threshold tuning is environment-specific
Behavioral/Anomaly ‚ñº

Detect deviations from established baseline.

Examples:

  • User accessing resources never accessed before
  • Process making network connections it normally doesn't
  • Login from new location/device

Pros:

Catches unknown threats

Cons:

Requires good baseline, higher false positives
Correlation-Based ‚ñº

Alert when multiple conditions occur together.

Examples:

  • Failed login + successful login from different IP
  • Suspicious email + macro execution
  • Privilege escalation + data access

Pros:

High fidelity, catches attack chains

Cons:

More complex to develop and maintain

Step 1: Identify Detection Need

  • Review MITRE ATT&CK coverage gaps
  • Threat intelligence on active campaigns
  • Post-incident lessons learned
  • Regulatory/compliance requirements
  • Red team/penetration test findings

Step 2: Research the Attack

  • What MITRE technique(s) does this cover?
  • What does the attack look like in logs?
  • What log sources are required?
  • What legitimate activity looks similar?
  • Are there existing detection rules to reference?

Step 3: Write the Detection Query

Query Best Practices
  • Start broad, then narrow with filters
  • Test against historical data first
  • Include relevant context in output
  • Use comments to explain logic
  • Consider performance (filter early)

Step 4: Tune and Deploy

  • Run in "audit mode" before alerting
  • Review sample results for false positives
  • Add exclusions for known legitimate activity
  • Set appropriate severity and priority
  • Enable alerting and monitor

Detection Examples

KQL Brute Force Detection (Threshold)
// T1110.001 - Password Guessing // Alert: Multiple failed logins followed by success SecurityEvent | where TimeGenerated > ago(1h) | where EventID in (4625, 4624) | summarize FailedCount = countif(EventID == 4625), SuccessCount = countif(EventID == 4624), SuccessTime = maxif(TimeGenerated, EventID == 4624) by TargetUserName, IpAddress, bin(TimeGenerated, 15m) | where FailedCount >= 5 and SuccessCount >= 1 | project TimeGenerated, TargetUserName, IpAddress, FailedCount, SuccessTime
KQL Suspicious PowerShell (Behavioral)
// T1059.001 - PowerShell // Alert: PowerShell with suspicious flags DeviceProcessEvents | where TimeGenerated > ago(1h) | where FileName =~ "powershell.exe" | where ProcessCommandLine has_any ( "-enc", "-encodedcommand", "-nop", "-noprofile", "-w hidden", "-windowstyle hidden", "bypass", "downloadstring", "invoke-expression", "iex" ) // Exclude known legitimate | where InitiatingProcessFileName !in~ ( "sccm.exe", "intune.exe" // Add your exclusions ) | project TimeGenerated, DeviceName, AccountName, ProcessCommandLine, InitiatingProcessFileName
KQL New Service Installation (Correlation)
// T1543.003 - Windows Service // Alert: New service with suspicious characteristics SecurityEvent | where TimeGenerated > ago(1h) | where EventID == 7045 // Service installed | extend ServiceName = tostring(EventData.ServiceName) | extend ServicePath = tostring(EventData.ImagePath) | where // Suspicious paths ServicePath has_any ("temp", "appdata", "public") or // Command execution in service ServicePath has_any ("cmd", "powershell", "wscript") | project TimeGenerated, Computer, ServiceName, ServicePath, SubjectUserName

Testing and Validation

Validation Methods

  • Historical testing: Run query against past data
  • Atomic Red Team: Execute simulated attacks
  • Purple team exercises: Coordinated attack simulation
  • Production monitoring: Track FP rate after deployment

Pre-Deployment Checklist

  • Query returns expected results for known attacks
  • False positive rate is acceptable (reviewed samples)
  • Performance is acceptable (query time reasonable)
  • Documentation is complete
  • Runbook/investigation steps are defined
  • Severity and priority are appropriate
Key Takeaways
  • Good alerts balance detection with false positive management
  • Test against historical data before enabling
  • Document everything‚Äîwhy, how, investigation steps
  • Start broad, then tune to reduce noise
  • Map detections to MITRE ATT&CK for coverage visibility
  • Include context in alert output for faster triage
  • Continuously tune based on analyst feedback
