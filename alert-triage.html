Triage Overview

Alert triage is the systematic process of evaluating and prioritizing security alerts to determine which require immediate attention, which can wait, and which are false positives. Effective triage is the cornerstone of SOC operationsâ€”it ensures critical threats are addressed quickly while managing alert fatigue.

The Goal of Triage

Make a rapid, accurate decision about an alert: Is this a true threat requiring action, a false positive, or something that needs deeper investigation? The key is speed with accuracy.

5-10 min
Target triage time
70-80%
Typical FP rate (untuned)
100%
Alerts need documentation

5-Step Triage Process

Follow this structured approach for every alert. Consistency is keyâ€”even experienced analysts can miss critical details when they skip steps.

Step 1: Read & Understand the Alert

Before doing anything else, thoroughly read what the alert is telling you.

  • What detection rule/signature triggered this alert?
  • What is the alert trying to detect? (Read the description)
  • What severity/confidence level was assigned?
  • What is the timestamp? Is this recent or old?
  • What are the key entities? (User, host, IP, file, process)
Pro Tip

If you don't understand what the alert is detecting, look it up in your wiki or MITRE ATT&CK before proceeding. You can't properly triage what you don't understand.

Step 2: Gather Context

Collect additional information about the entities involved in the alert.

For Users:

  • What is their role/department? (HR, IT admin, executive, contractor?)
  • Is this normal for their job function?
  • Have they been involved in recent alerts?
  • Are they on a watchlist or recently reported something?

For Hosts/Devices:

  • What type of system? (Workstation, server, DC, jump server?)
  • Who uses this system?
  • What is its criticality/sensitivity?
  • Is it compliant? (Patched, AV updated, etc.)

For IPs/Domains:

  • Is this internal or external?
  • What is the geolocation?
  • What is the reputation? (Check VirusTotal, AbuseIPDB)
  • Is it on known threat intel lists?

For Files/Hashes:

  • Is the file signed? By whom?
  • What is the VirusTotal score?
  • Has it been seen in the environment before?
  • What is the file path? (System directory vs. user temp folder)

Step 3: Check for Related Activity

One alert rarely tells the whole story. Look for correlated events.

  • Are there other alerts for the same user/host in the past 24-48 hours?
  • What other activity occurred on this host around the same time?
  • Are other hosts communicating with the same external IP?
  • Is this part of a larger campaign or pattern?
KQL Check related alerts for host
SecurityAlert | where TimeGenerated > ago(48h) | where CompromisedEntity == "HOSTNAME" or Entities contains "HOSTNAME" | project TimeGenerated, AlertName, Severity, Status | order by TimeGenerated desc

Step 4: Make a Determination

Based on your analysis, classify the alert and decide on action.

Classification Meaning Action
True Positive Real malicious activity confirmed Create incident, escalate, begin response
Suspicious Potentially malicious, needs more investigation Escalate to L2 for deeper analysis
False Positive Alert triggered but activity is benign Close with documentation, consider tuning
Benign True Positive Real activity detected but authorized Close, document exception, consider exclusion

Step 5: Document & Act

Every alert needs documentation regardless of outcome.

Documentation Requirements:

  • Summary of the alert and what triggered it
  • Investigation steps taken
  • Evidence/findings from your analysis
  • Classification decision with justification
  • Actions taken or recommended

Based on Classification:

  • True Positive: Create incident ticket, escalate, initiate response playbook
  • Suspicious: Escalate to L2 with findings, document what you've already checked
  • False Positive: Close alert, document root cause, submit tuning request if pattern repeats

Alert Classification

True Positive (TP) â–¼

Definition: The alert correctly identified malicious or unauthorized activity.

Examples:

  • Malware detected and confirmed to be malicious via sandbox/VT
  • Successful brute force with subsequent suspicious activity
  • Data exfiltration to known malicious infrastructure
  • Lateral movement indicators with confirmed compromise

Action: Immediate incident responseâ€”containment, investigation, remediation

Benign True Positive (BTP) â–¼

Definition: The alert correctly identified the activity, but it's authorized/expected.

Examples:

  • Penetration test activity (authorized)
  • IT admin using legitimate tools that trigger detections
  • Security tool scanning that triggers IDS alerts
  • Approved exception/deviation from policy

Action: Close alert, document authorization, consider exclusion rule

False Positive (FP) â–¼

Definition: The alert incorrectly flagged benign activity as malicious.

Examples:

  • Legitimate software flagged as malware
  • Normal admin activity matching attack patterns
  • Misconfigured detection rule
  • Environmental noise (backup jobs, inventory scans)

Action: Close alert, document reason, submit tuning request for repeat FPs

False Negative (FN) â–¼

Definition: Malicious activity occurred but no alert was generated.

Examples:

  • Attack technique not covered by detection rules
  • Logs not being collected from compromised system
  • Detection rule threshold too high
  • Evasion technique bypassed security controls

Action: Document gap, create detection rule request, review coverage

Prioritization Framework

When facing multiple alerts, prioritize based on these factors. The goal is to address the most critical and time-sensitive alerts first.

Priority Criteria Response Time Examples
P1 Critical Active threat, high-value target, confirmed compromise < 15 minutes Active ransomware, data exfiltration in progress, DC compromise
P2 High Confirmed malicious activity, significant risk < 1 hour Confirmed malware, successful phishing, account takeover
P3 Medium Suspicious activity, potential threat < 4 hours Unusual login patterns, policy violations, suspicious process
P4 Low Informational, minor policy violation < 24 hours Single failed login, low-risk policy deviation

Prioritization Factors

Asset Criticality

  • Domain Controller
  • Executive endpoint
  • Financial systems
  • Production servers
  • PII/sensitive data stores

Attack Stage

  • Active C2 communication
  • Lateral movement
  • Data exfiltration
  • Privilege escalation
  • Initial access/delivery

Alert Confidence

  • Multiple correlated alerts
  • High-fidelity detection rule
  • Matched threat intel IOC
  • EDR behavioral detection
  • Single low-confidence alert

Common Alert Patterns

Common False Positive Patterns

Alert Type Common FP Cause How to Verify
PowerShell encoded command SCCM, Intune, legitimate admin scripts Check parent process, scheduled task source
Suspicious process injection Security tools, AV, DLP agents Check if injecting process is known security tool
LSASS access Credential Guard, security software Check accessing process digital signature
Impossible travel VPN, proxy, mobile hotspot Check if IPs are known VPN/proxy services
Multiple failed logins Service account password change, cached creds Check if followed by successful login, service status

Red Flags - Likely True Positives

High Priority Indicators
  • Encoded PowerShell from Office application (Word, Excel)
  • Process spawning from suspicious parent (outlook.exe â†’ cmd.exe)
  • Communication with known C2 infrastructure
  • Credential dumping tool detected (mimikatz, procdump on lsass)
  • Ransomware file extension changes or ransom note creation
  • Large data transfers to new external destinations
  • Multiple failed logins followed by success from new location
  • Scheduled task or service creation by non-admin
  • Domain admin account used from workstation

Gray Areas - Need Investigation

These require additional context to determine if they're malicious:

  • Remote access tool (RMM) detected: Could be IT support or attacker persistence
  • Archive file with password: Could be legitimate or data staging
  • USB mass storage: Depends on policy and user role
  • Cloud storage upload: Business need or exfiltration?
  • After-hours activity: Dedicated employee or compromised account?

Triage Documentation

Every triage decision must be documented. Good documentation enables knowledge sharing, supports escalation, and provides audit trails.

Triage Note Template

## Alert Summary Alert Name: [Name of detection rule] Timestamp: [When alert fired] Severity: [Original severity] ## Affected Entities User: [username] - [role/department] Host: [hostname] - [type: workstation/server] IP/Domain: [if applicable] File/Hash: [if applicable] ## Investigation Steps 1. [What you checked first] 2. [Second check with findings] 3. [Additional context gathered] ## Findings - [Key finding 1] - [Key finding 2] - [Reputation check results] ## Classification [True Positive / False Positive / Benign TP / Escalate] Confidence: [High/Medium/Low] ## Justification [Why you made this determination] ## Actions Taken - [Action 1] - [Action 2] ## Analyst: [Your name] | Time Spent: [X minutes]
Documentation Anti-Patterns
  • Don't: "Checked, looks fine, closing" - No justification
  • Don't: "FP" - No investigation documented
  • Don't: Leave blank notes on closed alerts
  • Do: Document your reasoning even for obvious FPs
