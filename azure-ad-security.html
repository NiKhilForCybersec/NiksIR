Sign-in Logs

Azure AD sign-in logs are your primary data source for authentication monitoring. They capture every authentication attempt to Azure AD-integrated resources.

Key Sign-in Log Fields

Field Description Security Value
UserPrincipalName User's email/UPN Identify targeted account
IPAddress Source IP of authentication Location analysis, VPN detection
Location City, State, Country Impossible travel, unusual location
ResultType Error code (0 = success) Failed login analysis
AppDisplayName Application accessed Unusual app access
DeviceDetail Device info (browser, OS) New device detection
ConditionalAccessStatus CA policy result Policy bypass attempts
RiskLevelDuringSignIn Microsoft's risk assessment High-risk sign-ins
MfaDetail MFA method used MFA bypass attempts

Common Result Codes

Code Meaning Investigation
0 Success Normal login
50126 Invalid username/password Brute force, password spray
50053 Account locked Too many failed attempts
50057 Account disabled Attempt on disabled account
50074 MFA required User prompted for MFA
50076 MFA not completed MFA challenge failed/abandoned
53003 Blocked by CA policy Policy working as intended
50158 External security challenge Third-party MFA required

Risk Detection

Azure AD Identity Protection uses machine learning to detect risky sign-ins and users.

Risk Detection Types

Sign-in Risk (Real-time) ‚ñº
  • Anonymous IP: Sign-in from Tor, VPN, or anonymous proxy
  • Atypical travel: Impossible geographic movement
  • Malware-linked IP: IP associated with known malware
  • Unfamiliar sign-in properties: New device, location, etc.
  • Password spray: Multiple accounts, same password pattern
User Risk (Cumulative) ‚ñº
  • Leaked credentials: Found in dark web dumps
  • Azure AD threat intelligence: Microsoft's threat intel
  • Anomalous user activity: Unusual patterns

User risk persists until remediated (password reset, admin dismiss).

Risk Levels

Level Meaning Typical Response
High Strong indicator of compromise Block + force password reset
Medium Possible compromise Require MFA or block
Low Minor anomaly Monitor, may require MFA

Conditional Access

Conditional Access policies enforce access controls based on conditions. Understanding CA is essential for investigating blocked sign-ins and policy gaps.

Common CA Conditions

Who (Users)

  • All users
  • Specific groups
  • Guest users
  • Exclude break-glass accounts

What (Apps)

  • All cloud apps
  • Office 365
  • Specific applications
  • Microsoft Admin Portals

Where (Location)

  • Named locations (trusted)
  • Countries allowed/blocked
  • Corporate network

Controls

  • Require MFA
  • Require compliant device
  • Block access
  • Require password change
CA Policy Gaps

Common gaps: legacy authentication allowed, break-glass exclusions too broad, not all apps covered, guest access not restricted. Review CA policies during incident investigations.

Investigation Queries

KQL High-risk sign-ins
SigninLogs | where TimeGenerated > ago(24h) | where RiskLevelDuringSignIn in ("high", "medium") | project TimeGenerated, UserPrincipalName, RiskLevelDuringSignIn, RiskEventTypes, IPAddress, Location, AppDisplayName, ResultType, ResultDescription | order by TimeGenerated desc
KQL Failed logins by IP (password spray detection)
SigninLogs | where TimeGenerated > ago(1h) | where ResultType == "50126" | summarize TargetedAccounts = dcount(UserPrincipalName), AttemptCount = count(), Accounts = make_set(UserPrincipalName, 20) by IPAddress | where TargetedAccounts > 5 | order by TargetedAccounts desc
KQL Successful logins after multiple failures
let FailedLogins = SigninLogs | where TimeGenerated > ago(24h) | where ResultType != "0" | summarize FailCount = count() by UserPrincipalName, IPAddress; SigninLogs | where TimeGenerated > ago(24h) | where ResultType == "0" | join kind=inner FailedLogins on UserPrincipalName, IPAddress | where FailCount > 5 | project TimeGenerated, UserPrincipalName, IPAddress, FailCount, AppDisplayName, Location
KQL MFA fatigue / push spam
SigninLogs | where TimeGenerated > ago(24h) | where ResultType == "50074" // MFA required | summarize MfaPrompts = count() by UserPrincipalName, bin(TimeGenerated, 1h) | where MfaPrompts > 10 // Many prompts in one hour | order by MfaPrompts desc
KQL Legacy authentication attempts
SigninLogs | where TimeGenerated > ago(7d) | where ClientAppUsed in ("Exchange ActiveSync", "IMAP4", "POP3", "SMTP", "Authenticated SMTP", "Other clients") | summarize Count = count() by UserPrincipalName, ClientAppUsed, ResultType | order by Count desc

Response Actions

Account Remediation

Action When to Use How
Reset password Credentials compromised Azure Portal > Users > Reset password
Revoke sessions Active sessions may be hijacked Azure Portal > Users > Revoke sessions
Block sign-in Immediate containment needed Azure Portal > Users > Block sign-in
Require MFA re-registration MFA methods may be compromised Delete MFA methods, require re-registration
Confirm compromise Verified compromise (triggers CA) Identity Protection > Risky users > Confirm
Dismiss risk False positive confirmed Identity Protection > Risky users > Dismiss
PowerShell Revoke user sessions
# Connect to Microsoft Graph Connect-MgGraph -Scopes "User.ReadWrite.All" # Revoke all sessions for user Revoke-MgUserSignInSession -UserId "user@company.com" # Block sign-in Update-MgUser -UserId "user@company.com" -AccountEnabled:$false
Key Takeaways
  • Sign-in logs are primary source for authentication investigation
  • Result code 50126 = invalid credentials (brute force indicator)
  • High/Medium risk sign-ins need immediate investigation
  • Check for MFA fatigue (many prompts = potential attack)
  • Legacy authentication bypasses MFA‚Äîmonitor and block
  • Revoke sessions after password reset to kick out attackers
