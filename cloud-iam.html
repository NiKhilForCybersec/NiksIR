IAM Overview

Cloud Identity and Access Management (IAM) controls who can do what in cloud environments. Misconfigured IAM is the #1 cause of cloud breaches.

IAM Components

  • Identities: Users, groups, roles, service accounts
  • Policies: Permissions and access rules
  • Resources: What can be accessed
  • Actions: What can be done
  • Conditions: When/how access applies

Why IAM Matters

  • Controls all cloud access
  • Misconfiguration = breach
  • Complex and easy to get wrong
  • Privilege escalation paths
  • Shared responsibility model

Common IAM Risks

Risk Description Impact
Overprivileged Users Users with more permissions than needed Larger blast radius if compromised
Long-lived Access Keys Static credentials that never rotate Persistent access if leaked
Public Resources S3 buckets, storage accounts open to internet Data exposure
Cross-Account Trust Overly permissive role trust policies Unauthorized account access
Wildcard Permissions Policies with * actions or resources Unintended access
No MFA Root/admin accounts without MFA Easy account takeover

AWS IAM Monitoring

High-Risk AWS IAM Events

Event Risk Why It Matters
CreateUser Medium New identity created
CreateAccessKey High Persistent credential created
AttachUserPolicy High Privilege escalation
CreateRole Medium New assumable role
UpdateAssumeRolePolicy High Trust policy changed
PutBucketPolicy High S3 permissions changed
ConsoleLogin (root) Critical Root account usage
KQL AWS IAM privilege escalation events
AWSCloudTrail | where TimeGenerated > ago(24h) | where EventName in ( "CreateUser", "CreateAccessKey", "CreateLoginProfile", "AttachUserPolicy", "AttachRolePolicy", "PutUserPolicy", "AddUserToGroup", "UpdateAssumeRolePolicy" ) | project TimeGenerated, EventName, UserIdentityArn, SourceIpAddress, RequestParameters | order by TimeGenerated desc

Azure RBAC Monitoring

High-Risk Azure RBAC Events

Operation Risk Why It Matters
Create role assignment High Permissions granted
Add member to role High Privilege escalation
Create service principal Medium New app identity
Add app role assignment Medium App permissions changed
Add owner to application High Can modify app and secrets
Consent to application Medium OAuth consent grant

Privileged Azure AD Roles

  • Global Administrator: Full control of Azure AD
  • Privileged Role Administrator: Can assign any role
  • Application Administrator: Manage all apps
  • Cloud Application Administrator: Manage apps except App Proxy
  • Exchange Administrator: Full Exchange Online access

IAM Monitoring Best Practices

Key Detections

  • New admin/privileged role assignments
  • Access key creation for IAM users
  • Policy changes granting broad permissions
  • Root/Global Admin account usage
  • Cross-account role assumptions
  • Service principal credential additions
  • Unusual AssumeRole patterns

Baseline and Alert

What to Baseline Alert When
Who typically makes IAM changes Unusual user makes IAM change
Normal role assumption patterns Role assumed from new IP/region
Typical access key usage Key used from unusual location
Expected service principals New service principal created

Privilege Escalation Paths

Common Cloud Priv Esc Techniques

Attackers with limited cloud access often escalate privileges through IAM misconfigurations. Know these patterns to detect them.

AWS: iam:PassRole + Service Abuse ‚ñº

User can pass a privileged role to a service they control.

Example:

User with iam:PassRole and lambda:CreateFunction can create a Lambda that runs with admin role.

Detection:

Monitor PassRole events, especially passing admin roles.

AWS: CreateAccessKey for Others ‚ñº

User creates access keys for more privileged user.

Detection:

Alert when CreateAccessKey target differs from caller.

Azure: Application Owner Abuse ‚ñº

App owner can add credentials and use app's permissions.

Detection:

Monitor for new credentials added to high-privilege apps.

Cross-Account Role Assumption ‚ñº

Weak trust policy allows unauthorized role assumption.

Detection:

Monitor AssumeRole from unexpected accounts/principals.

Key Takeaways
  • IAM misconfiguration is #1 cloud breach cause
  • Monitor all privilege escalation events (policy attachments, role changes)
  • Access key creation is high-risk‚Äîpersistent credential
  • Root/Global Admin usage should be rare and alerted
  • Know privilege escalation paths (PassRole, app owner abuse)
  • Baseline IAM activity and alert on anomalies
  • Wildcard (*) permissions are dangerous‚Äîaudit regularly
