Cloud Service Models

Understanding service models helps determine who is responsible for what security controls.

IaaS

Infrastructure as a Service

You manage: OS, apps, data, network config

Provider manages: Hardware, virtualization, physical security

Examples: EC2, Azure VMs, GCE

PaaS

Platform as a Service

You manage: Apps, data

Provider manages: OS, runtime, middleware, infrastructure

Examples: Azure App Service, Elastic Beanstalk

SaaS

Software as a Service

You manage: User access, data, configuration

Provider manages: Everything else

Examples: M365, Salesforce, Workday

SOC Implication

The more you move toward SaaS, the less infrastructure you monitor but the more you rely on identity and application-level security. Always know what model you're dealing with.

Shared Responsibility Model

Security in the cloud is a shared responsibility between you and the cloud provider.

Responsibility Matrix

Component IaaS PaaS SaaS
Data & Content You You You
Identity & Access You You You
Application You You Provider
Network Controls You Shared Provider
Operating System You Provider Provider
Virtualization Provider Provider Provider

You Always Own

Data, identity, and access management are always your responsibility, regardless of service model. A misconfigured S3 bucket or overly permissive IAM role is on you, not AWS.

Major Cloud Providers

Amazon Web Services (AWS) ‚ñº

Largest cloud provider with the most services.

Key Security Services:

  • IAM: Identity and access management
  • CloudTrail: API activity logging (control plane)
  • GuardDuty: Threat detection service
  • Security Hub: Security posture management
  • VPC Flow Logs: Network traffic logging
  • Config: Resource configuration tracking

Key Log Sources:

CloudTrail, VPC Flow Logs, S3 Access Logs, GuardDuty Findings

Microsoft Azure ‚ñº

Integrates well with Microsoft ecosystem (M365, AD).

Key Security Services:

  • Entra ID (Azure AD): Identity management
  • Microsoft Defender for Cloud: CSPM and workload protection
  • Azure Activity Log: Control plane logging
  • Microsoft Sentinel: Cloud-native SIEM
  • Network Watcher: Network monitoring
  • Key Vault: Secrets management

Key Log Sources:

Activity Log, Sign-in Logs, Audit Logs, NSG Flow Logs

Google Cloud Platform (GCP) ‚ñº

Strong in data analytics, ML, and containerization.

Key Security Services:

  • Cloud IAM: Identity and access management
  • Cloud Audit Logs: Admin and data access logging
  • Security Command Center: Security posture
  • Chronicle: Security analytics (SIEM)
  • VPC Flow Logs: Network traffic logging
  • Cloud Armor: DDoS and WAF

Key Log Sources:

Admin Activity, Data Access, System Event, Policy Denied logs

Cloud-Specific Threats

Threat Description Detection
Misconfiguration Public S3 buckets, open security groups, weak IAM CSPM tools, Config auditing
Credential Compromise Stolen access keys, leaked secrets in code Unusual API calls, impossible travel
Privilege Escalation Exploiting IAM misconfig to gain admin Role assumption from unusual sources
Resource Hijacking Cryptomining using stolen credentials Unusual compute instances, high billing
Data Exfiltration Stealing data from storage services Unusual download patterns, new principals
Insecure APIs Exposed or vulnerable API endpoints API gateway logs, anomalous requests
Shadow IT Unauthorized cloud accounts/services CASB discovery, expense monitoring
Top Cloud Risks
  1. Misconfiguration - #1 cause of cloud breaches
  2. Excessive permissions - Over-privileged IAM roles
  3. Exposed credentials - Keys in code repos, logs

What to Monitor

High Priority

  • Root/admin account usage
  • IAM policy changes
  • Security group/firewall changes
  • New user/role creation
  • Failed authentication attempts
  • Public resource exposure
  • Encryption key access

Medium Priority

  • Unusual API activity
  • Resource creation in new regions
  • Large data transfers
  • Storage access patterns
  • Network flow anomalies
  • Configuration changes

Audit

  • All management plane operations
  • Billing anomalies
  • Resource inventory changes
  • Compliance violations
  • Third-party integrations

Key Takeaways
  • Understand the shared responsibility model‚Äîyou always own data and identity
  • IaaS = most responsibility, SaaS = least (but never zero)
  • Misconfiguration is the #1 cloud security risk
  • Enable and centralize cloud logging (CloudTrail, Activity Log, Audit Logs)
  • Monitor IAM changes and root/admin account usage closely
  • Use CSPM tools to continuously assess security posture
