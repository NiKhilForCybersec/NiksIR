Container Overview

Containers package applications with their dependencies, running as isolated processes. Understanding container architecture is essential for security monitoring.

Container Concepts

  • Image: Read-only template with app code
  • Container: Running instance of image
  • Registry: Store for container images
  • Orchestrator: Manages containers (K8s)
  • Pod: Smallest K8s unit (1+ containers)

Security Differences

  • Ephemeralâ€”containers come and go
  • Shared kernel with host
  • Immutable infrastructure model
  • Rapid deployment cycles
  • Dynamic networking

Container Threats

Vulnerable Images â–¼

Threat:

Images containing vulnerable packages or malicious code.

Indicators:

  • Images from untrusted registries
  • Base images with known CVEs
  • Images with embedded secrets

Detection:

Image scanning, registry security policies, admission controllers.

Container Escape â–¼

Threat:

Attacker breaks out of container to access host system.

Indicators:

  • Privileged container creation
  • Host path mounts
  • Capability escalation
  • Kernel exploit attempts

Detection:

Monitor privileged container launches, syscall anomalies, host filesystem access from containers.

Cryptomining â–¼

Threat:

Attackers deploy cryptominers in containers for profit.

Indicators:

  • High CPU usage containers
  • Connections to mining pools
  • Unknown images deployed

Detection:

Resource usage anomalies, network connections to known mining pools, unexpected container deployments.

Supply Chain Attacks â–¼

Threat:

Compromised dependencies or base images in build pipeline.

Indicators:

  • Unexpected changes in image layers
  • Images from unexpected sources
  • Unsigned images deployed

Detection:

Image signing verification, SBOM (Software Bill of Materials) analysis, pipeline security monitoring.

Key Detection Rules

  • Privileged container launched
  • Container running as root
  • hostPath or hostNetwork enabled
  • exec/attach to production pods
  • ClusterRole/ClusterRoleBinding changes
  • Secret access by unusual service accounts
  • Container with excessive capabilities
  • Image from unapproved registry

Investigation Techniques

Container Forensics Challenges

  • Ephemeral: Container may be gone before investigation
  • Immutable: Changes don't persist across restarts
  • Scale: Thousands of containers to monitor
  • Dynamic: Network and identity constantly changing

Investigation Commands

kubectl investigation 
# List all pods across namespaces
kubectl get pods --all-namespaces

# Get pod details
kubectl describe pod <pod-name> -n <namespace>

# Check pod logs
kubectl logs <pod-name> -n <namespace>

# Get events for troubleshooting
kubectl get events -n <namespace> --sort-by='.lastTimestamp'

# Check running processes in container
kubectl exec <pod-name> -n <namespace> -- ps aux

# Check network connections
kubectl exec <pod-name> -n <namespace> -- netstat -an

# List secrets (metadata only)
kubectl get secrets -n <namespace>

# Check RBAC for service account
kubectl auth can-i --list --as=system:serviceaccount:<ns>:<sa>

Action Command/Method

Capture for forensics Checkpoint container before deletion
