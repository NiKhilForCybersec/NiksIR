Why Documentation Matters

Documentation is not just administrative overhead‚Äîit's a critical security function. Good documentation enables effective handoffs, supports legal proceedings, improves detection rules, and helps the team learn from past incidents.

Continuity

Enables seamless handoffs between analysts and shifts

Legal/Compliance

Provides evidence trail for legal proceedings and audits

Improvement

Feeds into detection tuning and process improvements

Knowledge

Builds institutional knowledge for future reference

The 3 AM Test

Write documentation as if another analyst will need to pick up your work at 3 AM with no ability to contact you. Would they understand what you found, what you did, and what still needs to be done?

Documentation Principles

Principle Description Example
Be Specific Include exact values, timestamps, and identifiers "IP 192.168.1.50" not "a workstation"
Be Objective State facts, not assumptions or opinions "File was executed" not "attacker ran malware"
Be Complete Include negative findings too "No lateral movement detected" is valuable
Be Timely Document as you go, not after Update ticket in real-time during investigation
Be Structured Use consistent formats and templates Follow organizational templates
Be Searchable Include IOCs and key terms for future searches Include file hashes, IPs, domains in text

Ticket Documentation

Every alert investigation should be documented in a ticket with these elements:

Ticket Structure

1. Alert Summary ‚ñº
  • Alert Name: Exact alert title from SIEM/EDR
  • Alert ID: Unique identifier for reference
  • Severity: As assigned by detection system
  • Time: When the alert triggered (UTC)
  • Source: Which tool generated the alert
2. Affected Assets ‚ñº
  • Hostname: Device name(s) involved
  • IP Address: Internal and external IPs
  • User: Account(s) associated with activity
  • Asset Criticality: Business importance of affected systems
3. Investigation Summary ‚ñº
  • What was detected: Brief description of the activity
  • Analysis performed: Steps taken to investigate
  • Key findings: Important discoveries (positive and negative)
  • Evidence: Screenshots, query results, IOCs
4. Conclusion & Actions ‚ñº
  • Classification: True Positive, Benign True Positive, False Positive
  • Actions Taken: Containment, remediation steps
  • Recommendations: Follow-up actions, tuning suggestions
  • Escalation: If escalated, to whom and why

Classification Definitions

Classification Definition Example
True Positive (TP) Alert correctly identified malicious activity Malware detected and confirmed executed
Benign True Positive (BTP) Alert correctly fired but activity is authorized Penetration test, authorized admin tool
False Positive (FP) Alert fired incorrectly, no malicious activity Legitimate software flagged as malware

Investigation Notes

During complex investigations, maintain detailed running notes:

investigation_notes.md 
## Investigation: INC-2024-0142
## Analyst: jsmith | Started: 2024-01-15 14:30 UTC

### Initial Alert
- Alert: "Suspicious PowerShell Execution"
- Host: WKSTN-PC042 (10.1.5.42)
- User: mjohnson
- Time: 2024-01-15 14:22:15 UTC

### Timeline of Investigation

**14:32** - Reviewed alert in MDE
- PowerShell with encoded command detected
- Parent process: WINWORD.EXE
- Decoded command attempts to download from: hxxp://mal-domain[.]com/payload

**14:38** - Checked VirusTotal for domain
- mal-domain[.]com: 15/90 vendors flag as malicious
- Associated with Emotet campaign per community notes

**14:45** - Reviewed device timeline
- 14:20 - Email with attachment received
- 14:22 - Word document opened
- 14:22 - PowerShell spawned from Word
- 14:22 - Network connection blocked by firewall

**14:50** - Confirmed no successful download
- Firewall logs show connection blocked
- No additional processes spawned
- No persistence mechanisms found

### Findings
- Phishing email delivered malicious Word doc
- User opened document, enabling macros
- Payload download was blocked by firewall
- No evidence of successful compromise

### Actions Taken
- Isolated host as precaution
- Reset user password
- Blocked sender domain
- Submitted sample to sandbox

### Status: Contained - Pending user device scan
Investigation Note Tips
  • Timestamp every significant finding or action
  • Include exact queries used (can be collapsed/linked)
  • Note dead ends‚Äîwhat you checked that was negative
  • Use consistent formatting for IOCs (defang URLs)
  • Update status regularly for visibility

Incident Reports

Formal incident reports are required for confirmed security incidents.

Incident Report Structure

1. Executive Summary

Brief overview for management (2-3 paragraphs):

  • What happened (one sentence)
  • Impact and scope
  • Current status
  • Key actions taken

2. Incident Details

  • Incident ID and classification
  • Detection time and method
  • Affected systems and users
  • Attack vector and techniques (MITRE ATT&CK)

3. Timeline

Chronological sequence of events from initial access to containment

4. Impact Assessment

  • Data affected (type, volume, sensitivity)
  • Systems affected (count, criticality)
  • Business impact (downtime, financial)
  • Regulatory implications

5. Response Actions

  • Containment measures
  • Eradication steps
  • Recovery actions
  • Communication (internal/external)

6. Root Cause Analysis

How did the attack succeed? What controls failed?

7. Lessons Learned & Recommendations

  • What worked well
  • What could be improved
  • Specific recommendations with owners and timelines

8. Appendices

IOC list, detailed timeline, evidence inventory, query results

Key Takeaways
  • Document as you investigate, not after
  • Be specific, objective, and complete
  • Use consistent templates and formats
  • Include both positive and negative findings
  • Write for the analyst who will pick up your work at 3 AM
  • Good documentation improves detection and response over time
