Kernel vs User Mode

Kernel Mode Hooks

  • Minifilter drivers (file operations)
  • Callback routines (process/thread creation)
  • ETW providers (Event Tracing for Windows)
  • Network filter drivers

Advantage: Can't be bypassed by user-mode malware

User Mode Hooks

  • API hooking (ntdll.dll, kernel32.dll)
  • DLL injection for monitoring
  • Event log consumption
  • WMI subscriptions

Limitation: Can be bypassed by direct syscalls

Telemetry Sources

EDR collects telemetry from multiple sources. Understanding what's collected helps you know what you can and can't see.

Process Telemetry ‚ñº

What's Captured:

  • Process creation: PID, parent PID, command line, user, timestamp
  • Process termination: Exit code, duration
  • Process tree: Parent-child relationships
  • Image metadata: Hash, signature, version info, path
  • Token information: Privileges, integrity level

Detection Value:

Process trees reveal attack chains. Suspicious parent-child relationships (Word spawning PowerShell) are key indicators.

Blind Spots:

  • Process hollowing may show legitimate process name
  • Short-lived processes may be missed if sampling-based
File Telemetry ‚ñº

What's Captured:

  • File creation/modification/deletion: Path, timestamp, process
  • File reads: Often sampled, not exhaustive
  • File hashes: SHA256, SHA1, MD5
  • File metadata: Size, attributes, owner
  • Alternate data streams: Hidden data in NTFS

Detection Value:

Malware drops, persistence mechanisms, staging files, exfiltration preparation.

Blind Spots:

  • Memory-only malware leaves no file artifacts
  • High-volume file operations may be sampled
  • Encrypted containers hide content
Network Telemetry ‚ñº

What's Captured:

  • Connection events: Source/dest IP, ports, protocol
  • DNS queries: Queried domains, response IPs
  • Process attribution: Which process made the connection
  • Data volume: Bytes sent/received
  • URL/Host headers: For HTTP/HTTPS (if inspected)

Detection Value:

C2 communication, beaconing patterns, data exfiltration, lateral movement.

Blind Spots:

  • Encrypted traffic content (without SSL inspection)
  • DNS over HTTPS may bypass DNS logging
  • Traffic through legitimate services (cloud, CDN)
Registry Telemetry ‚ñº

What's Captured:

  • Key/value operations: Create, modify, delete
  • Persistence locations: Run keys, services, scheduled tasks
  • Security settings: Firewall, Defender, UAC changes

Key Locations Monitored:

  • HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
  • HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
  • HKLM\SYSTEM\CurrentControlSet\Services
  • HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon
Memory Telemetry ‚ñº

What's Captured:

  • Memory allocations: RWX (Read-Write-Execute) regions
  • Process injection: Cross-process memory writes
  • Module loading: DLLs loaded into processes
  • Memory scanning: Pattern matching for known malware

Detection Value:

Fileless malware, process injection, reflective DLL loading, shellcode.

Advanced Techniques Detected:

  • Process hollowing
  • DLL injection
  • Thread hijacking
  • APC injection

Behavioral Detection

Behavioral detection looks at what processes DO, not what they ARE. This catches novel malware that signatures miss.

Common Behavioral Indicators

Suspicious Process Chains ‚ñº

Examples:

  • Office ‚Üí cmd/PowerShell: Macro execution
  • Browser ‚Üí cmd/PowerShell: Drive-by download
  • WMI ‚Üí PowerShell: Lateral movement
  • Services ‚Üí cmd: Service exploitation
  • Explorer ‚Üí rundll32: User-initiated LOLBin

Why It Works:

Legitimate software rarely spawns command interpreters. When Word launches PowerShell, something is wrong.

Defense Evasion Behaviors ‚ñº
  • Disabling Windows Defender
  • Clearing event logs
  • Timestomping files
  • Process injection
  • Masquerading (renamed executables)
  • Indicator removal
Credential Access Behaviors ‚ñº
  • LSASS memory access
  • SAM database access
  • Credential file access (browser, SSH keys)
  • Kerberos ticket manipulation
  • DCSync replication requests
Persistence Behaviors ‚ñº
  • Registry Run key modification
  • Scheduled task creation
  • Service installation
  • WMI event subscription
  • Startup folder modification
  • DLL search order hijacking
Discovery Behaviors ‚ñº
  • Net commands (net user, net group, net share)
  • System information gathering (systeminfo, hostname)
  • Network enumeration (arp, netstat, route)
  • AD enumeration (nltest, dsquery)
  • File system enumeration

Evasion Awareness

Understanding how attackers evade EDR helps you investigate alerts more effectively and recognize sophisticated attacks.

Why This Matters

When investigating alerts, consider: What DIDN'T the EDR see? An attacker who knows EDR blind spots may have done more than what's visible.

Common Evasion Techniques

Technique How It Works Investigation Tip
Direct Syscalls Bypass user-mode API hooks by calling kernel directly Look for processes with unusual syscall patterns
Unhooking Remove EDR hooks from ntdll.dll Memory forensics can reveal unhooked DLLs
Process Hollowing Replace legitimate process memory with malware Compare on-disk vs in-memory image
Parent PID Spoofing Make malware appear spawned by legitimate parent Check for timing anomalies in process tree
Timestomping Modify file timestamps to blend in Compare $MFT timestamps vs $STANDARD_INFO
Living off the Land Use legitimate tools (LOLBins) for malicious purposes Context matters: why is certutil downloading?

LOLBins (Living off the Land Binaries)

Legitimate Windows binaries commonly abused by attackers:

Binary Legitimate Use Malicious Use
certutil.exe Certificate management Download files, decode base64
mshta.exe Run HTA applications Execute scripts, bypass AppLocker
regsvr32.exe Register COM objects Execute scripts via /s /u /i
rundll32.exe Run DLL functions Execute arbitrary code
bitsadmin.exe Background file transfers Download files, persist jobs
wmic.exe WMI queries Remote execution, reconnaissance

Alert Tuning

Effective EDR requires tuning to reduce false positives while maintaining detection of real threats.

Tuning Strategies

Strategy When to Use Risk
Exclude by path Known good software location Attackers may drop malware there
Exclude by hash Specific known-good file Lowest risk, very specific
Exclude by signer Trusted vendor software Signed malware, supply chain attacks
Suppress alert Alert is valid but not actionable May miss related malicious activity
Reduce severity Alert is noisy but worth keeping May be deprioritized when real

Tuning Best Practices

  • Document every exclusion - Why, when, who approved
  • Review exclusions periodically - Are they still needed?
  • Use narrowest scope possible - Hash > signer > path
  • Test before deploying - Validate in limited scope first
  • Monitor excluded items - Watch for abuse of exclusions
  • Never exclude temp folders - Common malware location
Key Takeaways
  • EDR uses kernel drivers + user-mode agents for comprehensive visibility
  • Process telemetry (parent-child chains) is most valuable for detection
  • Behavioral detection catches novel malware that signatures miss
  • LOLBins (certutil, mshta, rundll32) are commonly abused‚Äîcontext matters
  • Attackers evade EDR via direct syscalls, unhooking, process hollowing
  • When investigating, consider what EDR might NOT have seen
  • Tune with narrow exclusions and document everything
