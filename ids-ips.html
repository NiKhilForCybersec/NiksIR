IDS vs IPS

IDS - Intrusion Detection System

Passive monitoring

  • Monitors traffic passively (tap/span)
  • Generates alerts for suspicious activity
  • Does NOT block traffic
  • No impact on network performance

Use case: Visibility without risk of blocking legitimate traffic

IPS - Intrusion Prevention System

Inline blocking

  • Sits inline in traffic path
  • Can block malicious traffic
  • Real-time prevention
  • Potential latency impact

Use case: Active protection at network perimeter

Deployment Types

Type Location What It Sees
Network IDS/IPS (NIDS/NIPS) Network perimeter, segments All traffic passing through
Host IDS/IPS (HIDS/HIPS) On endpoint Traffic to/from that host
Wireless IDS/IPS Wireless network WiFi traffic and rogue APs
Modern Reality: NGFW

Most enterprises use Next-Generation Firewalls (NGFW) with integrated IPS functionality rather than standalone IDS/IPS. The concepts remain the same.

Detection Methods

Signature-Based Detection ‚ñº

Matches traffic against known attack patterns (signatures).

Pros:

  • Very accurate for known attacks
  • Low false positive rate when tuned
  • Fast processing

Cons:

  • Cannot detect unknown/zero-day attacks
  • Requires constant signature updates
  • Evasion possible with obfuscation
Anomaly-Based Detection ‚ñº

Establishes baseline of normal behavior, alerts on deviations.

Pros:

  • Can detect unknown attacks
  • Adapts to environment
  • Detects insider threats

Cons:

  • Higher false positive rate
  • Requires learning period
  • Baseline can be poisoned
Protocol Analysis ‚ñº

Validates traffic conforms to protocol specifications (RFCs).

Detects:

  • Malformed packets
  • Protocol violations
  • Buffer overflow attempts
  • Protocol tunneling
Behavioral Analysis ‚ñº

Analyzes patterns of behavior rather than individual events.

Detects:

  • Port scanning
  • Brute force attempts
  • Data exfiltration patterns
  • Lateral movement

Understanding Signatures

Signature Components

Component Description
Header Protocol, source/dest, ports
Content Payload patterns to match
Options Flow direction, thresholds, metadata
Action Alert, drop, pass, reject

Example Snort/Suricata Rule

snort.rules 
alert tcp $EXTERNAL_NET any -> $HOME_NET 445 (
    msg:"ET EXPLOIT MS17-010 EternalBlue Attempt";
    flow:to_server,established;
    content:"|00 00 00|";
    depth:3;
    content:"|ff|SMB";
    within:5;
    content:"|23 00 00 00 07 00 5c|PIPE|5c 00|";
    distance:0;
    sid:2024217;
    rev:3;
    classtype:attempted-admin;
)

Translation: Alert on TCP traffic from external to internal on port 445 that matches the EternalBlue exploit pattern in the SMB protocol.

Common Signature Categories

Category What It Detects
exploit Vulnerability exploitation attempts
malware Known malware signatures
policy Policy violations (P2P, Tor, etc.)
scan Reconnaissance/scanning activity
trojan Trojan/RAT communication
web-attack SQLi, XSS, web exploits
info Informational (not necessarily malicious)

Alert Analysis

Key Alert Fields

Field What to Check
Signature Name What attack/behavior was detected
Severity Vendor-assigned risk level
Source IP Who initiated the traffic
Destination IP Target of the traffic
Source/Dest Port Service involved
Protocol TCP, UDP, ICMP
Action Was traffic blocked or just alerted?
Payload/Packet What triggered the signature

Investigation Questions

  • Is source internal or external? External scanning vs internal compromise
  • Is target vulnerable? Does the target run the affected service/version?
  • Was traffic blocked? IPS block vs IDS alert only
  • Did follow-on activity occur? Did attack succeed?
  • Is this expected traffic? Vulnerability scanner, pentest?
  • Check source reputation Known malicious IP?
KQL Query IDS/IPS alerts from CommonSecurityLog
CommonSecurityLog | where TimeGenerated > ago(24h) | where DeviceVendor == "Palo Alto Networks" | where Activity == "THREAT" | project TimeGenerated, SourceIP, DestinationIP, DestinationPort, DeviceAction, ThreatDescription = Message, Severity = LogSeverity | order by TimeGenerated desc

Tuning & Optimization

Why Tune?

Out-of-the-box IDS/IPS generates too many alerts, many irrelevant to your environment. Tuning reduces noise and focuses on real threats.

Too Noisy

  • Alert fatigue
  • Real threats missed
  • Analyst burnout
  • SLA violations

Well Tuned

  • High-fidelity alerts
  • Manageable volume
  • Faster response
  • Better detection

Tuning Strategies

Strategy When to Use
Disable signature Not relevant to your environment (e.g., IIS attacks when you don't run IIS)
Threshold Alert only after N occurrences (reduce scan noise)
Suppress by source Known scanners, vulnerability assessment tools
Suppress by destination Honeypots, test systems that generate expected alerts
Time-based Suppress during scheduled maintenance windows
Severity adjustment Lower priority for low-risk targets
Document Tuning Decisions

Always document why a signature was disabled or suppressed. Future analysts need to understand the reasoning, and auditors may ask.

Key Takeaways
  • IDS monitors passively; IPS blocks inline
  • Signature-based is accurate but misses unknown attacks
  • Anomaly-based can detect unknowns but has more false positives
  • Check if target is vulnerable‚Äînot all exploits apply
  • Verify if traffic was blocked (IPS) or just alerted (IDS)
  • Tune aggressively to reduce noise and focus on real threats
