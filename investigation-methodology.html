Hypothesis-Driven Analysis

Don't investigate aimlessly. Form hypotheses about what might have happened, then systematically test them with evidence. This focuses your investigation and prevents rabbit holes.

Hypothesis Development Process

  1. Initial Hypothesis: Based on the alert, what's the most likely explanation?
  2. Alternative Hypotheses: What else could explain this activity?
  3. Evidence Needed: What would prove or disprove each hypothesis?
  4. Test: Gather and analyze evidence
  5. Refine: Adjust hypotheses based on findings
  6. Conclude: Which hypothesis best fits all evidence?

Example: Suspicious PowerShell Alert

Alert: Encoded PowerShell Execution ‚ñº

Initial Hypotheses:

Hypothesis Evidence That Would Support Evidence That Would Refute
H1: Malicious payload Unknown parent process, network connection to suspicious IP, persistence created Known management tool as parent, no network activity, recognized script
H2: IT admin script SCCM/Intune as parent, scheduled task, IT admin user context User-initiated, unusual parent process, non-IT user
H3: Legitimate software Signed parent process, known software behavior, consistent pattern Unsigned, first-time execution, user downloads folder

Investigation Path: Check parent process ‚Üí Decode the command ‚Üí Look for network connections ‚Üí Verify user context

Think Like an Attacker

Understanding attacker techniques helps you form better hypotheses. Ask yourself: "If I were an attacker, what would I do next?" This helps you know what evidence to look for.

Key Investigation Questions

Every investigation should answer these fundamental questions:

What Happened?

  • What activity triggered the alert?
  • What actions were performed?
  • What was the sequence of events?

When?

  • When did it start?
  • How long did it last?
  • Is it ongoing?

Where?

  • Which systems are affected?
  • Is it isolated or widespread?
  • Network location (internal/external)?

Who?

  • Which user account was involved?
  • Was it a legitimate user or compromised?
  • Is there an external threat actor?

How?

  • How did the attacker gain access?
  • What tools/techniques were used?
  • How did they move/persist?

Why? (Impact)

  • What was the objective?
  • Was data accessed or stolen?
  • What's the business impact?

Pivot Query Examples

KQL Pivot from suspicious IP to all affected hosts
let suspiciousIP = "203.0.113.50"; DeviceNetworkEvents | where TimeGenerated > ago(7d) | where RemoteIP == suspiciousIP | summarize FirstSeen = min(TimeGenerated), LastSeen = max(TimeGenerated), ConnectionCount = count(), Processes = make_set(InitiatingProcessFileName) by DeviceName, InitiatingProcessAccountName | order by FirstSeen asc
KQL Pivot from user to all systems accessed
let targetUser = "jsmith@contoso.com"; let timeRange = 24h; // Authentication events SigninLogs | where TimeGenerated > ago(timeRange) | where UserPrincipalName =~ targetUser | summarize Logins = count() by AppDisplayName, IPAddress, ResultType | union ( // Device logons DeviceLogonEvents | where TimeGenerated > ago(timeRange) | where AccountUpn =~ targetUser | summarize Logins = count() by DeviceName, LogonType, RemoteIP )

Timeline Reconstruction

Building a chronological timeline of events is essential for understanding the full attack chain and identifying the scope of compromise.

Timeline Building Process

  1. Anchor Event: Start with the alert or known malicious activity
  2. Expand Backward: What happened before? (Initial access, delivery)
  3. Expand Forward: What happened after? (Actions on objectives)
  4. Fill Gaps: Query additional data sources for missing periods
  5. Correlate: Align events from different sources by timestamp

Time Zone Awareness

Always be aware of time zones when correlating events. SIEM typically normalizes to UTC, but some logs may be in local time. Document your time zone assumptions.

Evidence Collection

Proper evidence collection supports your conclusions and enables further analysis if needed.

What to Collect

  • Screenshots of key findings
  • Exported query results
  • Relevant log excerpts
  • File hashes and samples
  • Network IOCs (IPs, domains)
  • Timeline documentation

How to Document

  • Include timestamps for all evidence
  • Note the source system/tool
  • Record your query or search terms
  • Explain the significance
  • Link to ticket/case number
  • Preserve chain of custody if needed
Evidence Preservation

For significant incidents, preserve evidence before taking remediation actions. Isolating a host or resetting credentials is important, but first ensure you've captured the forensic state needed for investigation.

Common Investigation Mistakes

Mistake Problem Better Approach
Tunnel Vision Focusing on first hypothesis, ignoring contradicting evidence Always consider alternative explanations
Incomplete Scope Only investigating the alerted host/user Pivot to find all affected systems
Premature Closure Closing investigation before full understanding Ensure all key questions are answered
Poor Documentation Not recording steps taken and evidence found Document as you go, not after
Not Asking for Help Struggling alone when stuck Escalate or consult peers early
Tool Fixation Only looking at one data source Correlate across multiple sources
Key Takeaways
  • Use hypothesis-driven analysis to focus your investigation
  • Answer the key questions: What, When, Where, Who, How, Why
  • Master pivoting to efficiently expand investigation scope
  • Build timelines to understand the full attack chain
  • Document everything‚Äîyour future self will thank you
  • Avoid common mistakes by staying systematic and open-minded
