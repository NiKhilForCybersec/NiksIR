Eradication Overview

Eradication is the process of removing the threat from the environment. This includes eliminating malware, removing persistence mechanisms, and ensuring the attacker can't regain access.

Complete Containment First

Before eradication, ensure the threat is fully contained. Eradicating from one system while the attacker still has access to others may alert them and cause escalation.

Eradication Goals

  • Remove all malware and tools
  • Eliminate all persistence mechanisms
  • Reset compromised credentials
  • Close the initial access vector
  • Verify threat is eliminated

Clean vs Rebuild

The fundamental decision: attempt to clean the system or rebuild from scratch?

Clean in Place

When to use:

  • Commodity malware (known, well-documented)
  • Single persistence mechanism identified
  • Confident in complete understanding
  • System is critical and hard to rebuild

Risks: May miss hidden persistence

Rebuild/Reimage

When to use:

  • APT or sophisticated attacker
  • Rootkit suspected
  • Unclear scope of compromise
  • Domain controller or critical server
  • Ransomware

Benefit: Guaranteed clean start

When in Doubt, Rebuild

If there's any uncertainty about the extent of compromise, rebuilding is safer. Advanced attackers use multiple persistence mechanisms and may have hidden backdoors.

Persistence Removal

Common Persistence Mechanisms

Registry Run Keys ‚ñº

Locations:

  • HKCU\Software\Microsoft\Windows\CurrentVersion\Run
  • HKLM\Software\Microsoft\Windows\CurrentVersion\Run
  • *\RunOnce, *\RunServices, *\RunServicesOnce

Removal:

# Check Run keys
reg query "HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run"
reg query "HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run"

# Delete specific value
reg delete "HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run" /v "MaliciousEntry" /f
Scheduled Tasks ‚ñº

Check:

# List all scheduled tasks
schtasks /query /fo LIST /v

# Check specific task
schtasks /query /tn "MaliciousTask" /fo LIST /v

Remove:

schtasks /delete /tn "MaliciousTask" /f
Services ‚ñº

Check:

# List services
sc query type= service state= all

# Check specific service
sc qc "MaliciousService"

Remove:

# Stop service
sc stop "MaliciousService"

# Delete service
sc delete "MaliciousService"
Startup Folder ‚ñº

Locations:

  • %APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup (User)
  • %ProgramData%\Microsoft\Windows\Start Menu\Programs\StartUp (All users)

Check and remove malicious files from these directories.

WMI Event Subscriptions ‚ñº

Check:

# List WMI subscriptions (PowerShell)
Get-WMIObject -Namespace root\subscription -Class __EventFilter
Get-WMIObject -Namespace root\subscription -Class __EventConsumer
Get-WMIObject -Namespace root\subscription -Class __FilterToConsumerBinding

Remove:

# Remove specific subscription
Get-WMIObject -Namespace root\subscription -Class __EventFilter | 
    Where-Object { $_.Name -eq "MaliciousFilter" } | Remove-WMIObject

Persistence Checklist

  • Registry Run/RunOnce keys
  • Scheduled tasks
  • Services
  • Startup folder items
  • WMI event subscriptions
  • Browser extensions
  • Office add-ins
  • DLL hijacking paths
  • COM object hijacking
  • AppInit_DLLs

Credential Reset

Any credentials that may have been exposed must be reset. The scope depends on what the attacker had access to.

Scenario Credentials to Reset
Single user workstation compromised That user's password, any saved passwords
Admin workstation compromised All admin accounts used from that machine
Lateral movement detected All accounts used on compromised systems
Domain controller compromised KRBTGT (twice), all privileged accounts, potentially all accounts
Credential dumping confirmed All accounts that logged into compromised systems
KRBTGT Reset Procedure

If domain controllers were compromised, reset KRBTGT password twice, with at least 10 hours between resets. This invalidates all Kerberos tickets. 

# Check current KRBTGT password last set
Get-ADUser krbtgt -Properties PasswordLastSet

# Reset (run twice, 10+ hours apart)
# Use Microsoft's KRBTGT reset script from Microsoft Security resources

Eradication Verification

Before moving to recovery, verify the threat is truly eliminated.

Verification Steps

  • Re-scan cleaned systems - Run full AV/EDR scans
  • Verify persistence removed - Check all known persistence locations
  • Monitor for re-infection - Watch cleaned systems closely
  • Check for C2 callbacks - Monitor network for beaconing
  • Review authentication logs - Watch for compromised credential use
  • Validate IOC blocks - Confirm all IOCs blocked at perimeter
KQL Monitor for reinfection indicators
let CleanedDevices = dynamic(["WORKSTATION-01", "WORKSTATION-02"]); let KnownBadHashes = dynamic(["abc123...", "def456..."]); DeviceFileEvents | where TimeGenerated > ago(24h) | where DeviceName in~ (CleanedDevices) | where SHA256 in (KnownBadHashes) | project TimeGenerated, DeviceName, FileName, FolderPath, SHA256
Key Takeaways
  • Complete containment before eradication
  • When in doubt, rebuild rather than clean
  • Check all persistence mechanisms‚Äîattackers use multiple
  • Reset all potentially exposed credentials
  • DC compromise requires KRBTGT reset (twice, 10+ hours apart)
  • Verify eradication before moving to recovery
  • Document all actions taken
