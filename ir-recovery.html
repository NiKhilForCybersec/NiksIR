Recovery Overview

Recovery is the process of restoring affected systems to normal operations. This phase focuses on bringing systems back online safely while preventing reinfection.

Recovery Goals

  • Restore system functionality
  • Return to normal operations
  • Validate security controls
  • Confirm threat elimination

Key Principles

  • Phased approach (critical first)
  • Verify before reconnecting
  • Enhanced monitoring
  • Document everything

Recovery Planning

Pre-Recovery Checklist

  • Eradication complete - Verified threat removed from all systems
  • Credentials reset - All compromised credentials changed
  • IOCs blocked - All indicators blocked at perimeter
  • Patches applied - Initial access vector closed
  • Backups validated - Clean backups identified (if needed)
  • Stakeholders notified - Business aware of recovery timeline

Recovery Priority Order

Priority System Type Examples
1. Critical Core infrastructure Domain controllers, DNS, DHCP, authentication
2. High Business critical Email, file servers, key applications
3. Medium Operational User workstations, printers, secondary apps
4. Low Non-critical Test systems, development environments

Execution Steps

1. System Restoration ‚ñº

For Reimaged Systems:

  • Deploy fresh OS image from known-good source
  • Apply all current patches before connecting to network
  • Install required applications
  • Restore user data from clean backups
  • Configure security agents (EDR, AV)

For Cleaned Systems:

  • Final verification scan before reconnection
  • Apply any pending patches
  • Verify security agents operational
2. Network Reconnection ‚ñº

Process:

  • Remove from isolation (EDR, VLAN)
  • Reconnect in phases, not all at once
  • Validate network connectivity
  • Confirm authentication working
  • Test application access
Staged Reconnection

Reconnect a few systems first and monitor for 24-48 hours before proceeding with the rest.

3. Service Validation ‚ñº

Verify:

  • Critical services running
  • Application functionality
  • User authentication
  • Data integrity
  • Integrations working
4. User Access Restoration ‚ñº
  • Notify users of password resets
  • Re-enable disabled accounts (if applicable)
  • Verify MFA functioning
  • Communicate any workflow changes
  • Provide support for access issues

Post-Recovery Monitoring

Enhanced monitoring is critical for 2-4 weeks after recovery to detect any missed persistence or reinfection.

Monitor For:

Reinfection Signs

  • Same or similar malware detected
  • Connections to blocked IOCs
  • Beaconing patterns resume
  • Credential abuse attempts

Operational Issues

  • Service failures
  • Authentication problems
  • Application errors
  • User access issues
KQL Monitor recovered systems for 30 days
let RecoveredDevices = dynamic(["WORKSTATION-01", "SERVER-01"]); union DeviceProcessEvents, DeviceNetworkEvents, DeviceFileEvents | where TimeGenerated > ago(30d) | where DeviceName in~ (RecoveredDevices) | where // Suspicious process indicators FileName has_any ("powershell", "cmd", "wscript") or RemoteIPType == "Public" or ActionType == "FileCreated" | summarize EventCount = count() by DeviceName, bin(TimeGenerated, 1d) | order by TimeGenerated desc

Monitoring Duration

Incident Severity Enhanced Monitoring Period
Commodity malware 1-2 weeks
Ransomware 2-4 weeks
Advanced threat/APT 4-8 weeks
Domain compromise 8+ weeks

Incident Closure

Once systems are stable and no reinfection is detected, the incident can be closed.

Closure Criteria

  • All affected systems restored
  • Enhanced monitoring period complete with no reinfection
  • Business operations returned to normal
  • All containment measures removed
  • Documentation complete
  • Lessons learned conducted

Post-Incident Activities

Documentation

  • Complete incident timeline
  • Actions taken at each phase
  • IOCs and detection signatures
  • Costs and impact assessment

Lessons Learned

  • What worked well?
  • What could be improved?
  • Detection gaps identified
  • Process improvements needed
Key Takeaways
  • Complete eradication before starting recovery
  • Recover in priority order (critical infrastructure first)
  • Phased reconnection with monitoring between phases
  • Enhanced monitoring for 2-4+ weeks post-recovery
  • Validate functionality before declaring systems operational
  • Document everything for lessons learned
  • Conduct lessons learned while details are fresh
