When to Isolate

Device isolation is a critical containment action that cuts off a compromised system from the network while maintaining EDR visibility.

Isolation Decision Matrix

Scenario Isolate? Rationale
Active ransomware execution âœ“ Yes - Immediately Prevent spread to network shares
Confirmed C2 communication âœ“ Yes - Immediately Cut attacker access
Active lateral movement âœ“ Yes - Immediately Contain spread
Credential theft detected âœ“ Yes Prevent credential use
Malware execution (contained) Consider If EDR blocked, may not need
Suspicious but unconfirmed Investigate first Avoid unnecessary business impact
PUP/adware detected âœ— No Low risk, handle without isolation
Critical server (production) Escalate first Business impact assessment needed
When in Doubt

If you're uncertain whether to isolate, escalate to a senior analyst or manager. For confirmed active threats (ransomware, C2), err on the side of isolationâ€”it's easier to release than to recover from spread.

Isolation Methods

Method How It Works Pros Cons
EDR Isolation Blocks network except EDR communication Quick, maintains visibility, reversible Requires EDR agent online
Network ACL/Firewall Block at network level by IP/MAC Works even if EDR compromised Loses all visibility
VLAN Quarantine Move to isolated network segment Controlled access for investigation Requires network team
Physical Disconnect Unplug network cable Guaranteed isolation Loses all remote access, last resort

EDR Isolation (Preferred)

What EDR Isolation Does:

  • Blocks all inbound and outbound network traffic
  • Allows communication only with EDR cloud backend
  • Maintains ability to query device and collect data
  • Allows remote response actions (live response)

What Still Works:

  • EDR telemetry collection
  • Remote investigation commands
  • File collection and analysis
  • Process termination

Isolation Procedure

Pre-Isolation Checklist

  • Confirm threat is real (not false positive)
  • Identify device owner and business impact
  • Document current device state (processes, connections)
  • Notify shift lead/manager if high-impact system
  • Prepare user communication

Isolation Steps

  1. Document the decision
    • Record why isolation is needed in ticket
    • Note time, analyst name, threat details
  2. Capture volatile data (if time permits)
    • Network connections (netstat)
    • Running processes
    • Current users
  3. Execute isolation in EDR console
    • Locate device
    • Select isolation action
    • Confirm action
  4. Verify isolation is active
    • Check device status shows isolated
    • Confirm EDR connectivity maintained
  5. Notify stakeholders
    • User (if reachable)
    • User's manager
    • Service desk
Speed vs Thoroughness

For active threats like ransomware: isolate first, document after. Every second of delay risks more damage. For less urgent situations, follow the full checklist.

User Communication

Notification Template

User Notification 
Subject: Security Alert - Your Computer Has Been Isolated

Hi [User Name],

Our security team detected potentially malicious activity on 
your computer ([Device Name]). As a precaution, we have 
temporarily isolated your device from the network.

What this means:
- Your computer cannot access network resources or the internet
- This is a protective measure, not an indication of wrongdoing
- Your files on the device are safe

What you should do:
- Do NOT shut down or restart your computer
- Do NOT attempt to reconnect to the network
- Contact the Security team at [phone/email] if you have questions

Next steps:
A security analyst will contact you shortly to discuss the 
situation and coordinate remediation.

Thank you for your patience.
Security Operations Team

Communication Tips

  • Be calm and reassuringâ€”users may panic
  • Avoid blame or accusatory language
  • Explain it's a protective measure
  • Give clear instructions (don't restart)
  • Provide contact information for questions

Post-Isolation Steps

Immediate Actions

  • Collect investigation package from EDR
  • Review process tree and timeline
  • Check for credential theft indicators
  • Hunt for same IOCs on other systems
  • Assess scopeâ€”are other systems affected?

Investigation Actions

  • Analyze malware sample if available
  • Identify initial access vector
  • Determine attacker actions on system
  • Check for persistence mechanisms
  • Assess data access/exfiltration

Password/Account Actions

If credentials may be compromised:

  • Reset user's password
  • Revoke active sessions
  • Re-register MFA if warranted
  • Check for unauthorized changes to account

Releasing Isolation

Release Criteria

  • Threat has been identified and eradicated
  • Malware removed/quarantined
  • Persistence mechanisms removed
  • No ongoing C2 communication
  • User credentials reset if compromised
  • Device passes security scan

Release Decision Authority

Scenario Who Can Release
False positive confirmed L1/L2 Analyst
Low-severity threat remediated L2 Analyst
High-severity incident L3/Manager approval
Critical system Manager + Business approval

Release Process

  1. Verify all remediation complete
  2. Document remediation actions taken
  3. Get approval if required
  4. Release isolation in EDR console
  5. Verify connectivity restored
  6. Notify user device is back online
  7. Monitor for recurrence (24-48 hours)
Key Takeaways
  • Isolate immediately for active threats (ransomware, C2, lateral movement)
  • EDR isolation is preferredâ€”maintains visibility and remote access
  • Document the decision and notify stakeholders
  • Communicate clearly with affected usersâ€”avoid blame
  • Don't release until threat is fully eradicated
  • Get appropriate approval before releasing high-severity isolations
  • Monitor for recurrence after release
