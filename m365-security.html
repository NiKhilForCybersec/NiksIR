Common M365 Threats

Account Compromise ‚ñº

Attacker gains access to M365 account via phishing or credential stuffing.

Indicators:

  • Inbox rules created (forwarding, hiding)
  • Unusual sign-in locations
  • Mass email sending
  • OAuth app consent grants
Inbox Rule Abuse ‚ñº

Attackers create rules to hide their activity or forward emails.

Suspicious Rules:

  • Forward all emails to external address
  • Delete emails containing "security" or "password"
  • Move emails to RSS Feeds or hidden folders
OAuth App Consent ‚ñº

User grants malicious app access to their data.

Risky Permissions:

  • Mail.Read / Mail.ReadWrite
  • Files.ReadWrite.All
  • User.Read.All

Response:

Revoke app consent, review activity
SharePoint/OneDrive Data Theft ‚ñº
  • Mass file downloads
  • Anonymous sharing links created
  • Sync to unmanaged device

Investigation Queries

KQL Emails with malware detected
EmailEvents | where TimeGenerated > ago(24h) | where ThreatTypes has "Malware" | project TimeGenerated, RecipientEmailAddress, SenderFromAddress, Subject, ThreatTypes, DeliveryAction, DeliveryLocation
KQL Inbox rules created
OfficeActivity | where TimeGenerated > ago(7d) | where Operation in ("New-InboxRule", "Set-InboxRule") | extend Parameters = tostring(Parameters) | where Parameters has_any ("ForwardTo", "RedirectTo", "DeleteMessage", "MoveToFolder") | project TimeGenerated, UserId, Operation, Parameters, ClientIP
KQL OAuth app consent grants
AuditLogs | where TimeGenerated > ago(7d) | where OperationName == "Consent to application" | mv-expand TargetResources | extend AppName = tostring(TargetResources.displayName) | extend Permissions = tostring( TargetResources.modifiedProperties) | project TimeGenerated, InitiatedBy, AppName, Permissions
KQL URL clicks to blocked sites
UrlClickEvents | where TimeGenerated > ago(24h) | where ActionType == "ClickBlocked" | project TimeGenerated, AccountUpn, Url, ThreatTypes, NetworkMessageId
KQL Mass file downloads from SharePoint
OfficeActivity | where TimeGenerated > ago(24h) | where Operation == "FileDownloaded" | where OfficeWorkload == "SharePoint" | summarize DownloadCount = count() by UserId, ClientIP | where DownloadCount > 50 | order by DownloadCount desc

Response Actions

Account Compromise Response

  • Reset password immediately
  • Revoke sessions (all active tokens)
  • Review inbox rules - delete malicious ones
  • Check OAuth apps - revoke unauthorized consents
  • Review sent items - was account used to spam?
  • Re-register MFA if methods may be compromised
  • Review mailbox audit log for data access

Removing Malicious Inbox Rules

PowerShell View and remove inbox rules
# Connect to Exchange Online Connect-ExchangeOnline # List all inbox rules for user Get-InboxRule -Mailbox "user@company.com" | Select-Object Name, Description, Enabled, ForwardTo, RedirectTo, DeleteMessage # Remove malicious rule Remove-InboxRule -Mailbox "user@company.com" -Identity "RuleName" # Disable forwarding completely Set-Mailbox -Identity "user@company.com" ` -ForwardingSmtpAddress $null ` -DeliverToMailboxAndForward $false
Key Takeaways
  • EOP is baseline protection; Defender for O365 adds advanced features
  • Check ZAP status‚Äîthreats may already be auto-remediated
  • Inbox rules are #1 persistence mechanism after compromise
  • OAuth consent grants can give attackers persistent access
  • After compromise: reset password, revoke sessions, check rules
  • Mass file downloads from SharePoint = potential data theft
