MFA Methods

Not all MFA is equal. Understanding the security properties of different methods helps prioritize alerts and assess risk.

Phishing-Resistant MFA

Phishing-resistant MFA methods (FIDO2, Windows Hello, certificates) cryptographically bind authentication to the legitimate site. Even if a user is phished, the attacker can't replay the authentication to the real site.

MFA Bypass Attacks

Real-Time Phishing (Adversary-in-the-Middle) â–¼

How It Works:

  1. Attacker creates phishing site that proxies to legitimate site
  2. User enters credentials on phishing site
  3. Attacker relays credentials to real site in real-time
  4. User receives MFA prompt, approves it
  5. Attacker captures session token/cookie

Tools Used:

Evilginx2, Modlishka, Muraena

Detection:

  • Sign-in from IP different than session IP
  • Unusual user-agent or client properties
  • Session token used from different location

Prevention:

Phishing-resistant MFA (FIDO2), Conditional Access policies

Session Hijacking / Token Theft â–¼

How It Works:

Attacker steals session token after successful MFA, bypassing need to authenticate again.

Methods:

  • Cookie theft from browser (infostealer malware)
  • Pass-the-Cookie attack
  • Token extraction from memory

Detection:

  • Session used from impossible travel locations
  • Session used from different device fingerprint
  • Activity after known compromise indicator
SIM Swapping â–¼

How It Works:

Attacker convinces carrier to transfer victim's phone number to attacker-controlled SIM, receiving all SMS MFA codes.

Indicators:

  • User reports phone not working
  • MFA codes sent but user didn't request
  • Account takeover shortly after phone issues

Prevention:

Don't use SMS MFA. If required, carrier PIN protection.

SS7 Attacks â–¼

How It Works:

Attackers exploit SS7 protocol vulnerabilities to intercept SMS messages without physical access to phone.

Requirements:

  • Access to SS7 network (telecom insider, nation-state)
  • Target phone number

Prevention:

Don't use SMS MFA for high-value accounts.

Help Desk Social Engineering â–¼

How It Works:

Attacker calls help desk pretending to be user, convinces them to reset MFA or add new authentication method.

Scenarios:

  • "I lost my phone, need to reset MFA"
  • "I'm locked out, urgent deadline"
  • "New employee, MFA not set up yet"

Prevention:

Strong identity verification before MFA changes. Callback to known number. Manager approval for resets.

MFA Fatigue Attacks

MFA Fatigue (Push Bombing)

Attacker repeatedly triggers MFA push notifications until the user accidentally or frustratedly approves one. High-profile breaches (Uber, Cisco) have used this technique.

MFA Fatigue Indicators

Indicator What to Look For
High volume prompts >5 MFA prompts in short period (minutes)
Off-hours attempts Late night/early morning when user might be groggy
Different source IPs Legitimate user location vs attacker location
Failed then success Many denials followed by approval
User complaints Reports of unexpected MFA prompts

Mitigations

  • Number matching: User must type number shown on screen
  • Location display: Show sign-in location in prompt
  • Rate limiting: Limit MFA prompts per time period
  • Lockout: Lock account after multiple denied prompts
  • Phishing-resistant MFA: FIDO2 eliminates push bombing

Detection Strategies

High-Priority MFA Alerts

Alert Risk Response
MFA method changed High Verify with user, check for compromise
New MFA method registered Medium-High Confirm user initiated
MFA reset by admin Medium Verify legitimate ticket/request
Many MFA prompts denied High MFA fatigue attack, contact user
MFA approved after many denials Critical Likely compromise, immediate response
Login from new location after MFA Medium Token theft possible

Investigation Queries

KQL MFA fatigue detection - many prompts in short time
SigninLogs | where TimeGenerated > ago(24h) | where ResultType == "50074" // MFA required or ResultType == "50076" // MFA denied | summarize MFAPrompts = count(), DeniedCount = countif(ResultType == "50076"), SourceIPs = make_set(IPAddress), TimeRange = max(TimeGenerated) - min(TimeGenerated) by UserPrincipalName, bin(TimeGenerated, 1h) | where MFAPrompts > 5 | order by MFAPrompts desc
KQL MFA method changes
AuditLogs | where TimeGenerated > ago(7d) | where OperationName has_any ( "User registered security info", "User changed default security info", "User deleted security info", "Admin registered security info" ) | extend TargetUser = tostring(TargetResources[0].userPrincipalName) | extend InitiatedBy = InitiatedBy.user.userPrincipalName | project TimeGenerated, OperationName, TargetUser, InitiatedBy, Result | order by TimeGenerated desc
KQL Successful login after MFA fatigue pattern
// Find users with many denied MFA then success let MFADenied = SigninLogs | where TimeGenerated > ago(24h) | where ResultType == "50076" // MFA denied | summarize DeniedCount = count(), FirstDenied = min(TimeGenerated) by UserPrincipalName, IPAddress | where DeniedCount > 3; SigninLogs | where TimeGenerated > ago(24h) | where ResultType == "0" // Success | join kind=inner MFADenied on UserPrincipalName | where TimeGenerated > FirstDenied | where TimeGenerated < FirstDenied + 2h | project TimeGenerated, UserPrincipalName, AttackerIP = IPAddress1, LoginIP = IPAddress, DeniedCount, AppDisplayName, Location
KQL Session anomaly after MFA - potential token theft
SigninLogs | where TimeGenerated > ago(24h) | where ResultType == "0" | summarize Locations = make_set(Location), IPs = make_set(IPAddress), UserAgents = make_set(UserAgent), LocationCount = dcount(Location), IPCount = dcount(IPAddress) by UserPrincipalName, CorrelationId | where LocationCount > 1 or IPCount > 1 | project UserPrincipalName, Locations, IPs, UserAgents

Response Actions

MFA Fatigue Attack Response

  1. Contact user immediately - Did they approve any prompts?
  2. If approved:
    • Reset password immediately
    • Revoke all sessions
    • Review sign-in logs for post-approval activity
    • Check for inbox rules, OAuth apps, persistence
  3. If not approved:
    • Still reset password (credentials are compromised)
    • Review where credentials were stolen
    • Consider temporary account lockout
  4. Block attacker IPs if identifiable
  5. Enable number matching if not already

MFA Reset Response

  • Verify reset was requested through legitimate channel
  • Check for help desk ticket or documented request
  • Confirm with user through known contact method
  • Review account for suspicious activity before/after reset
  • If unauthorized, treat as compromiseâ€”full investigation
Key Takeaways
  • Not all MFA is equalâ€”FIDO2/WebAuthn are phishing-resistant
  • SMS MFA is vulnerable to SIM swap and SS7 attacks
  • MFA fatigue attacks flood users with push notifications
  • Number matching significantly reduces fatigue attack success
  • Monitor for: many prompts, method changes, post-MFA anomalies
  • Real-time phishing can bypass non-phishing-resistant MFA
  • Session tokens can be stolen after successful MFA
