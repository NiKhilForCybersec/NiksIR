Forensics Fundamentals

Network forensics captures and analyzes network traffic to investigate security incidents, reconstruct attack timelines, and extract evidence.

When to Use Network Forensics

  • Investigating C2 communications
  • Analyzing data exfiltration
  • Reconstructing attack timeline
  • Identifying malware behavior
  • Legal evidence collection

Data Sources

  • Full packet capture (PCAP)
  • NetFlow/IPFIX metadata
  • Zeek/Bro connection logs
  • Firewall/proxy logs
  • DNS query logs

PCAP vs Flow Data

Aspect Full Packet Capture Flow Data (NetFlow)
Content Complete packet including payload Metadata only (IPs, ports, bytes)
Storage Very large (~1GB per minute at 1Gbps) Small (minutes per GB)
Retention Hours to days typically Months to years
Analysis Deep inspection, file extraction Traffic patterns, volume analysis
Use Case Incident investigation, malware analysis Baseline, anomaly detection, trending

Packet Capture

Capture Methods

Method Description Use Case
SPAN/Mirror Port Switch copies traffic to monitoring port Passive capture at network core
Network TAP Hardware device inline with traffic Full-duplex, no packet loss
Host-based Capture on endpoint (tcpdump, Wireshark) Specific host investigation
Firewall/Proxy Capture at security device Perimeter traffic

tcpdump Commands

tcpdump 
# Capture to file
tcpdump -i eth0 -w capture.pcap

# Capture specific host
tcpdump -i eth0 host 192.168.1.100 -w host.pcap

# Capture specific port
tcpdump -i eth0 port 443 -w https.pcap

# Capture with filter and limit
tcpdump -i eth0 'tcp port 80' -c 1000 -w web.pcap

# Capture between two hosts
tcpdump -i eth0 host 10.0.0.1 and host 10.0.0.2 -w conversation.pcap

# Read and display PCAP
tcpdump -r capture.pcap -nn

# Show packet contents (ASCII)
tcpdump -r capture.pcap -A

# Show packet contents (hex)
tcpdump -r capture.pcap -X

BPF Filter Syntax

Filter Description
host 192.168.1.1 Traffic to/from IP
src host 10.0.0.1 Traffic from IP
dst host 10.0.0.1 Traffic to IP
port 80 Traffic on port (either direction)
tcp TCP traffic only
udp port 53 DNS traffic
net 192.168.0.0/16 Traffic to/from subnet
not port 22 Exclude SSH
host A and host B Conversation between A and B

Wireshark Analysis

Essential Display Filters

Basic Filters ‚ñº
ip.addr == 192.168.1.1 Traffic to/from IP
ip.src == 10.0.0.1 Traffic from IP
tcp.port == 443 HTTPS traffic
tcp.flags.syn == 1 SYN packets (connection starts)
tcp.flags.reset == 1 RST packets (connection resets)
frame.time >= "2024-01-15 10:00:00" Time-based filter
HTTP Analysis ‚ñº
http All HTTP traffic
http.request HTTP requests only
http.response HTTP responses only
http.request.method == "POST" POST requests (data uploads)
http.host contains "evil" Requests to domains with "evil"
http.request.uri contains ".exe" Executable downloads
http.response.code == 200 Successful responses
DNS Analysis ‚ñº
dns All DNS traffic
dns.qry.name contains "evil" Queries for suspicious domains
dns.qry.type == 1 A record queries
dns.qry.type == 16 TXT record queries (may be tunneling)
dns.flags.response == 1 DNS responses
dns.count.answers > 10 Large DNS responses
TLS/SSL Analysis ‚ñº
tls All TLS traffic
tls.handshake TLS handshakes
tls.handshake.type == 1 Client Hello
tls.handshake.extensions_server_name contains "evil" SNI containing string
x509af.serialNumber Certificate serial numbers
Suspicious Traffic ‚ñº
tcp.analysis.retransmission Retransmissions (network issues)
tcp.analysis.zero_window Zero window (receiver overwhelmed)
icmp.type == 8 ICMP echo requests (ping)
tcp.port == 4444 Common Metasploit port
data.len > 1000 Large data payloads

Protocol Analysis

HTTP Investigation

What to Look For in HTTP
  • User-Agent: Unusual or known-malicious user agents
  • POST data: Encoded/encrypted payloads, credential theft
  • URLs: Downloads of executables, scripts, encoded paths
  • Response codes: 200s to suspicious domains, 404 beaconing
  • Headers: Custom headers used for C2 communication

DNS Investigation

Indicator Possible Meaning
Very long subdomain names DNS tunneling (data in DNS)
High volume TXT queries DNS tunneling or C2
Queries to non-standard TLDs Malicious infrastructure
Queries to IPs instead of domains Bypassing DNS logging
High entropy in subdomains DGA (Domain Generation Algorithm)
NXDOMAIN floods DGA beaconing attempts

SMB Investigation

SMB Red Flags
  • SMB connections between workstations (lateral movement)
  • Access to ADMIN$ or C$ shares
  • PsExec-style named pipe creation
  • Large file transfers via SMB
  • SMBv1 usage (vulnerable, should be disabled)

Evidence Extraction

File Extraction with Wireshark

  1. Open PCAP in Wireshark
  2. Go to File ‚Üí Export Objects
  3. Select protocol (HTTP, SMB, IMF, TFTP)
  4. Browse extracted files
  5. Save suspicious files for analysis

Manual Stream Extraction

tshark extraction 
# Extract HTTP objects
tshark -r capture.pcap --export-objects http,./extracted/

# Extract SMB objects
tshark -r capture.pcap --export-objects smb,./extracted/

# Get all HTTP requests
tshark -r capture.pcap -Y "http.request" -T fields \
  -e frame.time -e ip.src -e http.host -e http.request.uri

# Get all DNS queries
tshark -r capture.pcap -Y "dns.flags.response == 0" -T fields \
  -e frame.time -e ip.src -e dns.qry.name

Zeek for Automated Analysis

Zeek analysis 
# Process PCAP with Zeek
zeek -r capture.pcap

# Output files created:
conn.log      # All connections
http.log      # HTTP requests/responses
dns.log       # DNS queries
files.log     # File transfers detected
ssl.log       # TLS/SSL connections
weird.log     # Anomalies detected

# Query Zeek logs
cat conn.log | zeek-cut ts id.orig_h id.resp_h id.resp_p proto service
cat http.log | zeek-cut ts host uri user_agent
cat dns.log | zeek-cut ts query answers

Malware Traffic Analysis

Common Malware Traffic Patterns

C2 Beaconing ‚ñº

Characteristics:

  • Regular interval connections (jitter may be added)
  • Consistent destination (IP or domain)
  • Similar request sizes
  • Often uses HTTP/HTTPS to blend in

Wireshark Filter:

http.request && ip.dst == [suspicious_ip]
Data Exfiltration ‚ñº

Characteristics:

  • Large outbound transfers
  • POST requests with large bodies
  • DNS tunneling (encoded data in queries)
  • Uploads to cloud storage

Wireshark Filter:

http.request.method == "POST" && tcp.len > 1000
Lateral Movement ‚ñº

Characteristics:

  • SMB traffic between workstations
  • WinRM/PSRemoting connections
  • RDP to unusual systems
  • Pass-the-hash authentication patterns

Wireshark Filter:

smb || tcp.port == 5985 || tcp.port == 3389
Payload Download ‚ñº

Characteristics:

  • Executable downloads (.exe, .dll, .ps1)
  • Encoded/obfuscated payloads
  • Downloads from unusual domains
  • certutil, PowerShell, bitsadmin user-agents

Wireshark Filter:

http.request.uri contains ".exe" || http.request.uri contains ".dll"

Investigation Checklist

  • Identify source and destination of suspicious traffic
  • Determine timeline of communications
  • Extract any transferred files
  • Analyze protocols and ports used
  • Look for patterns (beaconing intervals, data volumes)
  • Check TLS certificates for anomalies
  • Correlate with endpoint logs
  • Document all findings with timestamps
Key Takeaways
  • PCAP provides full content; flow data provides long-term metadata
  • tcpdump captures; Wireshark analyzes; Zeek automates parsing
  • Follow TCP Stream to see complete conversations
  • Export Objects to extract transferred files
  • Look for beaconing patterns: regular intervals, consistent destinations
  • DNS tunneling: long subdomains, high TXT queries, high entropy
  • SMB between workstations often indicates lateral movement
  • Always correlate network findings with endpoint telemetry
