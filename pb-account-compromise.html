L1 L2 PB-04 Response Playbook

Playbook Overview

Trigger

Suspicious sign-in, impossible travel, credential theft indicator

Default Severity

P2 High

Target SLA

Response: 30 min | Resolution: 4 hours

Escalation

L1 ‚Üí L2 (if privileged account or data access)

  • Investigation Steps

Account Compromise Indicators

Signs that may indicate an account has been compromised:

High Confidence

  • Login from known malicious IP
  • Credentials found in phishing kit
  • Password in breach database + recent use
  • User reports not making activity
  • MFA bypass detected
  • Session token theft evidence

Medium Confidence

  • Impossible travel alert
  • Login from unusual location
  • After-hours access (unusual for user)
  • Multiple failed then success
  • New device + sensitive access
  • Email forwarding rules added

Lower Confidence

  • VPN/proxy usage
  • New browser or OS
  • Password change
  • Single unusual login
  • Legacy auth usage

Immediate Actions

If Compromise is Confirmed or Highly Likely
  1. Reset password immediately
  2. Revoke all active sessions/tokens
  3. Verify/re-register MFA
  4. Check for persistence mechanisms (email rules, OAuth apps)
  5. Contact user through alternate channel (not email)
PowerShell Revoke Azure AD sessions (Microsoft Graph)
# Connect to Microsoft Graph Connect-MgGraph -Scopes "User.ReadWrite.All" # Revoke all refresh tokens for user Revoke-MgUserSignInSession -UserId "user@domain.com" # Reset password (generates random password) $password = @{ Password = "TempP@ss123!" ForceChangePasswordNextSignIn = $true } Update-MgUser -UserId "user@domain.com" -PasswordProfile $password

Investigation Steps

Step 1: Review Sign-in Activity ‚ñº

Examine all authentication events for the account.

KQL Azure AD sign-in history
let targetUser = "user@domain.com"; SigninLogs | where TimeGenerated > ago(30d) | where UserPrincipalName =~ targetUser | project TimeGenerated, ResultType, ResultDescription, IPAddress, Location, DeviceDetail, AppDisplayName, ClientAppUsed, ConditionalAccessStatus, RiskState, RiskLevelDuringSignIn | order by TimeGenerated desc

Key Questions:

  • Any logins from unusual locations or IPs?
  • Any logins during unusual hours?
  • Any new devices or browsers?
  • Any failed attempts before success?
  • Any legacy authentication (non-MFA)?
Step 2: Check for Persistence ‚ñº

Attackers establish persistence to maintain access even after password reset.

Common Persistence Mechanisms:

Mechanism What to Check How to Check
Email Forwarding Rules Rules forwarding email externally Exchange admin, Outlook rules
OAuth App Consents Malicious apps with mail/file access Azure AD Enterprise Apps, user consents
MFA Methods Attacker-added phone/authenticator Azure AD user authentication methods
Mailbox Delegates Full access granted to other accounts Exchange mailbox permissions
Device Registration Attacker device joined to Azure AD Azure AD user devices
KQL Check for mailbox rule changes
let targetUser = "user@domain.com"; OfficeActivity | where TimeGenerated > ago(30d) | where UserId =~ targetUser | where Operation in ( "New-InboxRule", "Set-InboxRule", "Set-Mailbox", "Add-MailboxPermission", "Set-MailboxFolderPermission" ) | project TimeGenerated, Operation, Parameters, ClientIP
KQL Check for OAuth app consents
let targetUser = "user@domain.com"; AuditLogs | where TimeGenerated > ago(30d) | where OperationName == "Consent to application" | where InitiatedBy.user.userPrincipalName =~ targetUser | extend AppName = TargetResources[0].displayName | project TimeGenerated, AppName, TargetResources
Step 3: Review Account Activity ‚ñº

Determine what the attacker did with the compromised account.

KQL Email activity (sent, read, deleted)
let targetUser = "user@domain.com"; let suspiciousTime = datetime(2024-01-15T10:00:00Z); OfficeActivity | where TimeGenerated > suspiciousTime | where UserId =~ targetUser | where Operation in ("Send", "SendAs", "SendOnBehalf", "MailItemsAccessed", "MoveToDeletedItems") | project TimeGenerated, Operation, ClientIP, MailboxOwnerUPN, DestinationMailboxOwnerUPN | order by TimeGenerated asc
KQL File access (SharePoint/OneDrive)
let targetUser = "user@domain.com"; OfficeActivity | where TimeGenerated > ago(7d) | where UserId =~ targetUser | where RecordType in ("SharePointFileOperation", "OneDrive") | where Operation in ("FileDownloaded", "FileAccessed", "FileCopied", "FileModified") | summarize FileCount = count(), Files = make_set(SourceFileName) by Operation, ClientIP | order by FileCount desc

Key Activities to Review:

  • Email: Sent phishing internally? Accessed sensitive emails?
  • Files: Downloaded sensitive documents?
  • Admin Actions: Any privilege escalation attempts?
  • Applications: Accessed other connected apps?
Step 4: Identify Attack Vector ‚ñº

Determine how the account was compromised to prevent recurrence.

Attack Vector Indicators Investigation
Phishing User clicked link, credentials entered Check email logs, user interview
Credential Stuffing Password in breach database Check HaveIBeenPwned, password age
Password Spray Many users targeted, common password Check org-wide failed logins
Token Theft Session continues after password change Adversary-in-the-middle, malware
Malware Infostealer on user device EDR investigation on device

Scope Assessment

Determine if other accounts were compromised or affected.

  • Check if same attacker IP accessed other accounts
  • Check if compromised user sent phishing internally
  • Check if attack is part of larger campaign
  • Review recipients of any emails sent by attacker
  • Check for shared credentials (same password elsewhere)
KQL Find other accounts accessed from attacker IP
let attackerIP = "203.0.113.50"; SigninLogs | where TimeGenerated > ago(30d) | where IPAddress == attackerIP | where ResultType == "0" // Successful logins | summarize FirstSeen = min(TimeGenerated), LastSeen = max(TimeGenerated), LoginCount = count(), Apps = make_set(AppDisplayName) by UserPrincipalName | order by FirstSeen asc

Remediation Checklist

  • Password: Reset to strong unique password
  • Sessions: Revoke all active sessions and tokens
  • MFA: Re-register MFA methods (remove unknown methods)
  • Email Rules: Remove malicious inbox/forwarding rules
  • OAuth Apps: Revoke suspicious app consents
  • Devices: Remove unknown registered devices
  • Delegates: Remove unauthorized mailbox permissions
  • Block Attacker: Block attacker IP if possible
  • User Education: Brief user on what happened and how to avoid
  • Monitor: Enhanced monitoring for 30 days

Privileged Account Handling

Elevated Response for Admin Accounts

Compromised admin accounts require immediate escalation and expanded investigation.

Additional Steps for Admin Accounts:

  • Immediately escalate to IR team/management
  • Check all admin actions during compromise window
  • Look for new accounts created
  • Check for privilege escalation to other accounts
  • Review Azure AD/AD configuration changes
  • Check for Global Admin or Exchange Admin actions
  • Review Conditional Access policy changes
  • Consider assuming full environment compromise
KQL Admin actions by compromised account
let targetUser = "admin@domain.com"; let compromiseStart = datetime(2024-01-15T10:00:00Z); AuditLogs | where TimeGenerated > compromiseStart | where InitiatedBy.user.userPrincipalName =~ targetUser | project TimeGenerated, OperationName, Result, TargetResources, AdditionalDetails | order by TimeGenerated asc
Playbook Summary
  1. Contain: Reset password, revoke sessions, verify MFA
  2. Investigate: Review sign-ins, check persistence, review activity
  3. Scope: Check for other compromised accounts
  4. Remediate: Remove all attacker persistence
  5. Root Cause: Identify how credentials were compromised
  6. Document: Record findings and actions taken
