L1 L2 PB-06 Response Playbook

Playbook Overview

Trigger

Multiple failed logins, password spray detection

Default Severity

P3 Medium (escalate if successful)

Response SLA

Response: 2 hours | Resolution: 8 hours

Escalation

Escalate if any account compromised

Playbook Steps
  • Attack Types
  • Detection Indicators
  • Investigation Steps
  • Response Actions
  • Prevention Measures

Attack Types

Traditional Brute Force

Many passwords â†’ One account

  • Attacker tries many passwords against single account
  • Usually triggers account lockout
  • Easier to detect

Detection: High failed logins for one user

Password Spray

One password â†’ Many accounts

  • Attacker tries common password across many accounts
  • Avoids account lockout
  • Harder to detect

Detection: Same password error across accounts

Credential Stuffing

Leaked credentials â†’ Your accounts

  • Uses credentials from other breaches
  • Exploits password reuse
  • Often from botnets

Detection: Failed logins from many IPs

Detection Indicators

Attack Type Indicators Log Sources
Brute Force
  • >10 failed logins for same account
  • Rapid succession (seconds apart)
  • May trigger lockout
 Azure AD, AD (4625), VPN, SSH
Password Spray
  • Same error code across multiple accounts
  • From same IP or small IP range
  • 1-2 attempts per account
  • Spread over time (hours)
 Azure AD, AD, M365
Credential Stuffing
  • Many IPs, many accounts
  • Residential/proxy IPs
  • Varied user agents
  • Attempts against user list
 WAF, Load balancer, App logs
KQL Detect password spray pattern
SigninLogs | where TimeGenerated > ago(24h) | where ResultType == "50126" // Invalid username or password | summarize FailedAccounts = dcount(UserPrincipalName), AccountList = make_set(UserPrincipalName), AttemptCount = count() by IPAddress, bin(TimeGenerated, 1h) | where FailedAccounts > 10 // Many accounts from same IP | order by FailedAccounts desc
KQL Detect brute force against single account
SigninLogs | where TimeGenerated > ago(24h) | where ResultType != "0" // Failed logins | summarize FailedCount = count(), IPs = make_set(IPAddress), ErrorCodes = make_set(ResultType) by UserPrincipalName, bin(TimeGenerated, 1h) | where FailedCount > 15 | order by FailedCount desc

Investigation Steps

Step 1: Assess Attack Scope â–¼
  • How many accounts targeted?
  • How many source IPs involved?
  • What time period?
  • What services targeted (Azure AD, VPN, OWA, RDP)?
Step 2: Check for Successful Logins â–¼

CRITICAL: Determine if any attack succeeded.

KQL Check for success after failed attempts
let suspiciousIP = "203.0.113.50"; SigninLogs | where TimeGenerated > ago(24h) | where IPAddress == suspiciousIP | where ResultType == "0" // Successful login | project TimeGenerated, UserPrincipalName, AppDisplayName, Location, DeviceDetail
If Successful Login Found

Escalate to P2 High and follow Account Compromise Playbook

Step 3: Analyze Source IPs â–¼
  • Check IP reputation (AbuseIPDB, VirusTotal)
  • Determine geolocation
  • Identify if it's known proxy/VPN/Tor
  • Check if IP has accessed other resources

Password sprays often come from cloud hosting or residential proxies

Step 4: Identify Targeted Accounts â–¼
  • Are specific users targeted (executives, admins)?
  • Is it a full user list (enumeration)?
  • Are the targeted accounts valid or mostly invalid?
  • Any pattern in targeted usernames?

Response Actions

If Attack is Unsuccessful (Blocked)

  • Block attacker IP(s) at firewall/WAF/Conditional Access
  • Review account lockout policies (are they working?)
  • Check if targeted accounts need password reset anyway
  • Document attack details for threat intelligence
  • Consider reporting to abuse contacts if applicable

If Attack is Successful (Compromised Account)

Immediate Actions
  1. Reset compromised account password immediately
  2. Revoke all active sessions
  3. Re-verify MFA methods
  4. Block attacker IP(s)
  5. Follow Account Compromise Playbook

Blocking at Conditional Access (Azure AD)

PowerShell Add IP to named location block list
# This is typically done via Azure Portal or Graph API # Create Named Location with blocked IPs # Then reference in Conditional Access policy to block # Example: Check current sign-in activity from IP Get-MgAuditLogSignIn -Filter "ipAddress eq '203.0.113.50'" -Top 50

Block Legacy Authentication

Legacy authentication protocols (POP, IMAP, SMTP AUTH) don't support MFA and are frequently targeted. Block them via Conditional Access unless specifically needed.

Playbook Summary
  1. Identify: What type of attack? Scope?
  2. Check Success: Did any login succeed? (Critical)
  3. Analyze: Source IPs, targeted accounts
  4. Block: Attacker IPs at appropriate layer
  5. Remediate: Reset any compromised accounts
  6. Harden: Verify MFA, block legacy auth
