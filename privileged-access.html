PAM Overview

Privileged Access Management (PAM) protects and monitors accounts with elevated permissions. For SOC analysts, understanding PAM helps detect privilege abuse and insider threats.

80%
Breaches involve privileged credentials
74
Days average dwell time
#1
Target: Domain Admins

PAM Core Functions

Credential Vaulting

Store privileged credentials securely, rotate automatically

Just-in-Time Access

Grant privileges only when needed, revoke after

Session Recording

Record admin sessions for audit and investigation

Access Control

Approve/deny privileged access requests

Privileged Account Types

Account Type Examples Risk Level Monitoring Priority
Domain Admins DA accounts, Enterprise Admins Critical Every login, every action
Local Admins Local administrator accounts High Lateral movement detection
Service Accounts SQL, IIS, backup service accounts High Interactive logon, credential use
Cloud Admins Global Admin, Azure Owner Critical All privileged operations
Application Admins Database admins, app owners Medium Data access, config changes
Break-Glass Accounts Emergency access accounts Critical Any use = alert
Service Account Risk

Service accounts are high-value targets: they often have broad access, rarely change passwords, and don't use MFA. Interactive logon by a service account is a red flag.

Monitoring Strategies

What to Monitor

Authentication Events ‚ñº
  • Admin account logins (especially after hours)
  • Logins from unusual locations/IPs
  • Failed login attempts
  • Service account interactive logons
  • Break-glass account use
Privilege Changes ‚ñº
  • Users added to privileged groups
  • Role assignments in Azure AD
  • Permission changes on critical resources
  • GPO modifications
  • New admin accounts created
Privileged Actions ‚ñº
  • Password resets by admins
  • Security policy changes
  • Audit log modifications
  • Backup/restore operations
  • Domain controller access

PAM Solution Alerts

If your organization uses PAM (CyberArk, BeyondTrust, Thycotic, etc.), monitor:

  • Credential checkouts (who, when, for what)
  • Session recordings (review for suspicious activity)
  • Failed access requests
  • Bypass attempts
  • Orphaned sessions (not checked back in)

Detecting Privilege Abuse

Red Flags

üö© High Confidence

  • Break-glass account used without ticket
  • Domain Admin from non-PAW system
  • Service account interactive logon
  • Admin activity during vacation

üö© Medium Confidence

  • Admin login from new location
  • After-hours privileged activity
  • Unusual number of password resets
  • Access to resources outside normal scope

üö© Investigate

  • New admin account created
  • Self-elevation to privileged group
  • Admin accessing non-admin resources
  • Pattern change in admin behavior

Investigation Queries

KQL Admin logons from non-admin workstations
let AdminAccounts = dynamic(["admin1", "admin2"]); // Or query from table let AdminWorkstations = dynamic(["PAW-01", "PAW-02"]); SecurityEvent | where TimeGenerated > ago(24h) | where EventID == 4624 | where TargetUserName in (AdminAccounts) | where Computer !in (AdminWorkstations) | project TimeGenerated, TargetUserName, Computer, LogonType, IpAddress
KQL Service account interactive logons
SecurityEvent | where TimeGenerated > ago(7d) | where EventID == 4624 | where LogonType in (2, 10) // Interactive, RemoteInteractive | where TargetUserName matches regex "^svc_" // Service account naming or TargetUserName matches regex "_svc$" | project TimeGenerated, TargetUserName, Computer, LogonType, IpAddress
KQL After-hours admin activity
let AdminAccounts = dynamic(["admin1", "admin2"]); SecurityEvent | where TimeGenerated > ago(7d) | where EventID == 4624 | where TargetUserName in (AdminAccounts) | extend Hour = datetime_part("hour", TimeGenerated) | where Hour < 6 or Hour > 20 // Before 6am or after 8pm | project TimeGenerated, TargetUserName, Computer, IpAddress
KQL Azure AD privileged role assignments
AuditLogs | where TimeGenerated > ago(7d) | where OperationName has "Add member to role" | mv-expand TargetResources | extend RoleName = tostring(TargetResources.displayName) | where RoleName has_any ("Global Administrator", "Privileged Role Administrator", "Security Administrator") | extend InitiatedBy = tostring(InitiatedBy.user.userPrincipalName) | project TimeGenerated, InitiatedBy, RoleName, Result
Key Takeaways
  • 80% of breaches involve privileged credentials
  • Service account interactive logon = immediate investigation
  • Break-glass account use should always trigger alert
  • Admins should only log in from dedicated workstations (PAWs)
  • Monitor for after-hours privileged activity
  • Track all privileged group membership changes
  • Correlate PAM checkouts with actual activity
