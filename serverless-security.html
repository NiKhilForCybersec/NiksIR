Serverless Overview

Serverless computing runs code without managing servers. Functions execute on-demand and scale automatically. Security focus shifts from infrastructure to application and identity.

Serverless Platforms

Platform Service Log Source
AWS Lambda CloudWatch Logs, CloudTrail
Azure Functions Application Insights, Activity Log
GCP Cloud Functions Cloud Logging

Security Differences

What You Don't Manage

  • OS patching
  • Runtime security
  • Network infrastructure
  • Scaling infrastructure

What You Must Secure

  • Function code
  • Dependencies
  • IAM permissions
  • Environment variables
  • Triggers and inputs

Serverless Threats

Overprivileged Functions â–¼

Threat:

Functions with excessive IAM permissions become privilege escalation vectors.

Example:

Lambda with iam:* can create admin users if compromised.

Detection:

  • Audit function IAM roles
  • Alert on sensitive API calls from functions
  • Monitor for unexpected resource access
Injection Attacks â–¼

Threat:

Untrusted input leads to command injection, SQL injection, etc.

Attack Path:

Attacker sends malicious payload via API Gateway â†’ Lambda executes command.

Detection:

  • Monitor function errors and exceptions
  • Look for unusual commands in logs
  • WAF alerts on injection patterns
Secrets in Environment Variables â–¼

Threat:

Credentials stored in environment variables can be extracted.

Risk:

Anyone with function read access can view environment variables.

Detection:

  • Audit GetFunction/GetFunctionConfiguration calls
  • Scan for secrets in function configs
  • Alert on unauthorized access to function details
Vulnerable Dependencies â–¼

Threat:

Third-party packages with vulnerabilities or malicious code.

Example:

Compromised npm package in Lambda layer.

Detection:

  • Dependency scanning in CI/CD
  • Monitor for unexpected network connections
  • Function behavior anomalies
Event Injection â–¼

Threat:

Attacker triggers function with malicious event data.

Triggers at Risk:

  • S3 events (malicious file upload)
  • API Gateway (crafted requests)
  • SNS/SQS (message manipulation)

Detection:

Monitor trigger sources, validate event data patterns.

Monitoring Strategies

Key Events to Monitor

Event Risk Why It Matters
CreateFunction Medium New code deployed
UpdateFunctionCode Medium Code changed
UpdateFunctionConfiguration Medium Settings/env vars changed
AddPermission High Who can invoke changed
GetFunction Low-Med Code/config retrieved
Function errors spike Medium May indicate attack

Log Sources

  • CloudTrail/Activity Log: API calls (create, update, invoke)
  • CloudWatch/Application Insights: Function execution logs
  • X-Ray/Distributed Tracing: Request flow, latency
  • WAF Logs: API Gateway attack attempts

Investigation

Function Compromise Investigation

  1. Review function changes: When was code/config last modified?
  2. Check IAM activity: What did the function's role do?
  3. Analyze invocations: Unusual triggers or patterns?
  4. Review logs: Errors, unusual output, commands?
  5. Check network: Unexpected outbound connections?
  6. Assess data access: What resources did it touch?
KQL Lambda function changes
AWSCloudTrail | where TimeGenerated > ago(7d) | where EventSource == "lambda.amazonaws.com" | where EventName in ( "CreateFunction", "UpdateFunctionCode20150331v2", "UpdateFunctionConfiguration20150331v2", "AddPermission20150331v2" ) | project TimeGenerated, EventName, UserIdentityArn, SourceIpAddress, RequestParameters | order by TimeGenerated desc

Response Actions

Action When to Use
Disable function Active compromise, stop execution
Revoke IAM role Stop function's cloud access
Remove triggers Prevent invocations
Rotate secrets If env vars may be exposed
Rollback code Restore known-good version
Key Takeaways
  • Serverless shifts security focus to code and identity
  • Overprivileged functions are the biggest risk
  • Monitor function changes and permission modifications
  • Don't store secrets in environment variables
  • Validate all inputâ€”functions are attack surfaces
  • Error spikes may indicate injection attempts
  • Response includes disabling function and revoking IAM
