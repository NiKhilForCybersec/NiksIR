Service Account Overview

Service accounts are non-human identities used by applications, services, and automated processes. They're essential for operations but often represent significant security risk due to their privileges and lack of oversight.

Types of Service Accounts

Type Description Risk Level
Domain Service Accounts AD accounts for services (svc_*, svc-*) High
Managed Service Accounts (MSA) AD-managed, auto-rotating passwords Medium
Group Managed Service Accounts (gMSA) MSA for multiple servers Medium
Local System/Service Built-in Windows accounts (NT AUTHORITY\SYSTEM) Varies
Cloud Service Principals Azure AD app registrations, AWS IAM roles High
API Keys / Tokens Non-interactive authentication credentials High

Why Service Accounts Are Different

User Accounts

  • Interactive logon expected
  • MFA can be enforced
  • Regular password changes
  • Conditional access policies
  • User behavior analytics

Service Accounts

  • Non-interactive (usually)
  • MFA often not possible
  • Passwords rarely changed
  • Often excluded from policies
  • Baseline is predictable

Security Risks

The Service Account Problem

Service accounts often have broad permissions, rarely-changed passwords, no MFA, and minimal monitoring. They're a favorite target for attackers because compromising one can provide persistent, privileged access.

Common Service Account Issues

Excessive Privileges â–¼

Problem:

Service accounts granted Domain Admin or overly broad permissions "because it works."

Risk:

  • Compromise gives attacker domain-wide access
  • Lateral movement without additional credential theft

Detection:

Audit service accounts in privileged groups. Alert on service accounts accessing resources outside their normal scope.

Stale/Unchanged Passwords â–¼

Problem:

Passwords never rotated due to fear of breaking applications.

Risk:

  • Compromised passwords remain valid indefinitely
  • More time for password cracking attacks
  • Kerberoasting becomes more viable

Detection:

Monitor pwdLastSet attribute. Alert on accounts with passwords older than policy allows.

Interactive Logon â–¼

Problem:

Service accounts used for interactive logon (RDP, console).

Risk:

  • Strong indicator of compromise or misuse
  • Credentials cached on endpoint

Detection:

Alert on any Event 4624 LogonType 2 or 10 for service accounts. This should be rare/never.

SPNs and Kerberoasting â–¼

Problem:

Service accounts with SPNs have tickets requestable by any domain user.

Risk:

  • Offline password cracking of service account
  • No failed login alerts generated
  • Works against any account with SPN

Detection:

Monitor Event 4769 for service ticket requests, especially with RC4 encryption (0x17) which is easier to crack.

Orphaned Accounts â–¼

Problem:

Service accounts for decommissioned applications still exist and active.

Risk:

  • Unknown attack surface
  • No owner to notice misuse
  • Permissions may still be broad

Detection:

Compare service account inventory against active services. Alert on dormant accounts that suddenly become active.

Common Attacks

Attack MITRE ATT&CK Description Detection
Kerberoasting T1558.003 Request service tickets, crack offline 4769 with RC4 encryption, many TGS requests
Password Spraying T1110.003 Try common passwords across accounts 4625 from same source to multiple accounts
Credential Theft T1003 Dump service account credentials from memory/disk LSASS access, service account in memory dump
Token Impersonation T1134.001 Steal service account token from process Token manipulation events, unusual service account activity
Persistence via Service T1543.003 Create service running as privileged account New service creation, service modification

Detection Strategies

High-Fidelity Alerts

Detection Why It's High Fidelity
Interactive logon by service account Service accounts should never log in interactively
Service account used from non-standard host Service accounts typically run from specific servers
Service account activity outside business hours Automated services run on predictable schedules
Service account password change Rarely changesâ€”could indicate compromise
Service account added to privileged group Privilege escalation attempt

Baseline-Based Detection

Service accounts are ideal for baseline detection because their behavior is predictable. Establish baselines for:

  • Source hosts the account authenticates from
  • Target resources the account accesses
  • Time of day activity occurs
  • Volume of activity (connections, queries)
  • Protocols used (Kerberos vs NTLM)

Investigation Queries

KQL Interactive logons by service accounts
// Service account naming convention: svc_*, svc-*, _svc SecurityEvent | where TimeGenerated > ago(7d) | where EventID == 4624 | where LogonType in (2, 10) // Interactive, RemoteInteractive | where TargetUserName matches regex @"^svc[_-]|_svc$" or TargetUserName has "service" | project TimeGenerated, Computer, TargetUserName, LogonType, IpAddress, LogonProcessName | order by TimeGenerated desc
KQL Service accounts authenticating from new hosts
// Baseline: hosts seen in last 30 days let ServiceAccounts = SecurityEvent | where TimeGenerated > ago(30d) | where EventID == 4624 | where TargetUserName matches regex @"^svc[_-]" | summarize KnownHosts = make_set(Computer) by TargetUserName; // Find new hosts in last 24 hours SecurityEvent | where TimeGenerated > ago(1d) | where EventID == 4624 | where TargetUserName matches regex @"^svc[_-]" | join kind=inner ServiceAccounts on TargetUserName | where Computer !in (KnownHosts) | project TimeGenerated, TargetUserName, Computer, IpAddress, LogonType | summarize Count = count() by TargetUserName, Computer
KQL Kerberoasting detection - service ticket requests with RC4
SecurityEvent | where TimeGenerated > ago(24h) | where EventID == 4769 // TGS request | where TicketEncryptionType == "0x17" // RC4 | where ServiceName !endswith "$" // Exclude computer accounts | where ServiceName != "krbtgt" | summarize RequestCount = count(), TargetServices = make_set(ServiceName), ServiceCount = dcount(ServiceName) by IpAddress, TargetUserName | where ServiceCount > 3 // Multiple services targeted | order by ServiceCount desc
KQL Service account password age check
// Requires IdentityInfo table (Azure AD or ingested AD data) IdentityInfo | where AccountName matches regex @"^svc[_-]" | extend PasswordAge = datetime_diff('day', now(), PasswordLastSet) | where PasswordAge > 90 // Older than 90 days | project AccountName, PasswordLastSet, PasswordAge, IsAccountEnabled, GroupMembership | order by PasswordAge desc
KQL Service account added to privileged group
let PrivilegedGroups = dynamic([ "Domain Admins", "Enterprise Admins", "Schema Admins", "Administrators", "Backup Operators", "Account Operators" ]); SecurityEvent | where TimeGenerated > ago(7d) | where EventID in (4728, 4732, 4756) // Member added to group | where TargetUserName in (PrivilegedGroups) | where MemberName matches regex @"svc[_-]|_svc|service" | project TimeGenerated, Computer, TargetUserName, MemberName, SubjectUserName

Best Practices

Inventory and Governance

  • Maintain inventory of all service accounts
  • Document owner, purpose, and required permissions
  • Regular review (quarterly) of necessity and permissions
  • Decommission unused accounts promptly

Security Controls

  • Use gMSA where possible (automatic password rotation)
  • Apply principle of least privilege
  • Disable interactive logon (Deny log on locally)
  • Restrict logon to specific hosts (Log on to)
  • Use strong, unique passwords (25+ characters)
  • Rotate passwords regularly (90 days or less)

Monitoring Requirements

  • Alert on interactive logons
  • Alert on authentication from unexpected hosts
  • Alert on activity outside normal hours
  • Monitor for Kerberoasting (RC4 ticket requests)
  • Alert on privilege changes
  • Track password changes
Key Takeaways
  • Service accounts are high-value targetsâ€”privileged, rarely monitored
  • Interactive logon by service account = immediate investigation
  • Kerberoasting targets accounts with SPNsâ€”monitor Event 4769 with RC4
  • Baseline service account behaviorâ€”deviations are suspicious
  • Use gMSA for automatic password rotation where possible
  • Inventory, document, and regularly review all service accounts
  • Apply principle of least privilege and restrict logon hosts
