What is SIEM?

Security Information and Event Management (SIEM) is the central nervous system of a SOC. It collects, aggregates, and analyzes log data from across the enterprise to detect security threats, support investigations, and meet compliance requirements.

SIEM Core Functions

Collect ‚Üí Parse ‚Üí Normalize ‚Üí Correlate ‚Üí Alert ‚Üí Store

Security Information (SI)

Long-term storage, analysis, and reporting of log data for compliance, forensics, and trend analysis.

Event Management (EM)

Real-time monitoring, correlation, and alerting on security events for immediate threat detection.

Why SIEM Matters

  • Centralized Visibility: Single pane of glass across all security data sources
  • Threat Detection: Correlation rules identify attacks that span multiple systems
  • Incident Investigation: Rapid search and analysis during security incidents
  • Compliance: Log retention and reporting for regulatory requirements
  • Forensics: Historical data for post-incident analysis

Data Flow Pipeline

1. Collection ‚ñº

Raw logs are gathered from data sources across the environment.

Key Data Sources for SOC:

  • Endpoints: Windows Event Logs, Sysmon, EDR telemetry
  • Network: Firewall, IDS/IPS, proxy, DNS, VPN
  • Identity: Active Directory, Azure AD, authentication systems
  • Cloud: AWS CloudTrail, Azure Activity Logs, M365 audit logs
  • Applications: Web servers, databases, custom apps
  • Security Tools: AV, DLP, email gateway, CASB
2. Parsing ‚ñº

Raw log data is broken down into structured fields.

# Raw Log Jan 15 10:23:45 webserver sshd[12345]: Failed password for admin from 192.168.1.100 port 22 ssh2 # Parsed Fields { "timestamp": "2024-01-15T10:23:45", "host": "webserver", "process": "sshd", "pid": 12345, "event_type": "authentication_failure", "user": "admin", "source_ip": "192.168.1.100", "port": 22, "protocol": "ssh2" }
3. Normalization ‚ñº

Different log formats are mapped to a common schema for consistent analysis.

Source Original Field Normalized Field
Windows IpAddress src_ip
Firewall srcaddr
Linux SRC

Common schemas include: OCSF (Open Cybersecurity Schema Framework), ECS (Elastic Common Schema), CIM (Splunk Common Information Model)

4. Enrichment ‚ñº

Additional context is added to logs to improve analysis.

Common Enrichment Types:

  • GeoIP: Add location data to IP addresses
  • Asset Information: Add hostname, owner, criticality from CMDB
  • User Context: Add department, role, manager from HR systems
  • Threat Intelligence: Flag known malicious IPs, domains, hashes
  • DNS Resolution: Add hostname for IP or IP for hostname
5. Indexing & Storage ‚ñº

Processed logs are indexed for fast searching and stored for retention.

  • Hot Storage: Recent data (days-weeks), fast SSD, expensive
  • Warm Storage: Medium-term (weeks-months), slower, moderate cost
  • Cold/Archive: Long-term (months-years), slowest, cheapest

Correlation & Detection

Correlation is where SIEM transforms raw logs into actionable security alerts.

Single-Event Rules

Alert on individual events matching criteria.

Example: Alert when malware detected by AV

Threshold Rules

Alert when count exceeds threshold in time window.

Example: >10 failed logins in 5 minutes

Sequence Rules

Alert on specific sequence of events.

Example: Failed logins ‚Üí successful login ‚Üí sensitive file access

Anomaly Detection

ML-based detection of deviation from baseline.

Example: User accessing 10x normal number of files

Alerting & Thresholds

Alert Severity Levels

Severity Criteria Response
Critical Confirmed active threat, critical system impact Immediate investigation, potential IR activation
High Likely malicious activity, significant risk Investigate within 30 minutes
Medium Suspicious activity, potential threat Investigate within SLA (typically 2-4 hours)
Low Informational, minor policy violation Review during normal operations

Tuning Best Practices

Reducing Alert Fatigue
  • Start with conservative thresholds and tighten based on data
  • Create exceptions for known legitimate activity
  • Use asset criticality to adjust severity
  • Regularly review and retire low-value alerts
  • Track false positive rates per rule

Storage Cost Considerations

SIEM costs often scale with data volume (GB/day ingested). Work with your team to understand which logs provide security value versus noise. Not all logs need real-time indexing‚Äîsome can go to cold storage for compliance only.

Key Takeaways
  • SIEM centralizes security data from across the enterprise for detection and investigation
  • Data flows through: Collection ‚Üí Parsing ‚Üí Normalization ‚Üí Enrichment ‚Üí Indexing
  • Correlation rules transform raw logs into actionable security alerts
  • Alert tuning is ongoing‚Äîbalance detection coverage with alert fatigue
  • Retention policies balance security needs, compliance, and cost
