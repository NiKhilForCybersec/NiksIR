Why Tune

SIEM tuning reduces alert fatigue, improves analyst efficiency, and ensures real threats aren't buried in noise. Untuned environments lead to missed detections and burnout.

Untuned SIEM Problems

  • Alert fatigueâ€”analysts ignore alerts
  • Real threats buried in noise
  • Wasted analyst time on false positives
  • SLA breaches due to volume
  • Low team morale

Well-Tuned SIEM Benefits

  • Actionable, high-fidelity alerts
  • Faster response to real threats
  • Efficient use of analyst time
  • Consistent SLA achievement
  • Better team engagement
Tuning is Continuous

Tuning isn't a one-time activity. Environments change, new software is deployed, and attacker techniques evolve. Build tuning into regular SOC operations.

False Positive Analysis

Identifying False Positives

FP Source Examples Tuning Approach
Legitimate Software Admin tools, deployment systems, security scanners Exclude by process/path/user
Business Processes Batch jobs, automated reports, scheduled tasks Exclude by schedule or service account
Threshold Too Low Normal activity exceeds threshold Raise threshold based on baseline
Overly Broad Query Detection matches too much Add specificity to query logic
Environment-Specific Normal for your org, not others Custom exclusions

FP Analysis Process

  1. Collect samples: Review 10-20 recent alerts
  2. Categorize: Group by reason for false positive
  3. Identify pattern: What makes these benign?
  4. Design exclusion: Narrow, specific exclusion
  5. Test: Verify exclusion doesn't hide real threats
  6. Document: Record why exclusion was added

Tuning Techniques

Threshold Adjustment â–¼

Adjust count/time thresholds based on environment baseline.

Example:

Failed login alert fires at 5 failures. Analysis shows normal users often have 8-10 failures. Adjust to 15 to reduce noise while still catching attacks.

// Before: Too sensitive | where FailedCount >= 5 // After: Based on baseline analysis | where FailedCount >= 15
Entity Exclusions â–¼

Exclude specific users, hosts, IPs, or processes.

Example:

// Exclude known service accounts | where AccountName !in~ ( "svc_backup", "svc_monitor", "svc_deploy" ) // Exclude IT admin workstations | where DeviceName !startswith "IT-ADMIN-" // Exclude known scanner IPs | where IPAddress !in ( "10.0.1.100", "10.0.1.101" // Vuln scanners )
Time-Based Exclusions â–¼

Exclude alerts during known maintenance windows or batch jobs.

Example:

// Exclude during backup window (2-4 AM) | where not ( hourofday(TimeGenerated) >= 2 and hourofday(TimeGenerated) < 4 and AccountName == "svc_backup" ) // Exclude during maintenance window | where not ( dayofweek(TimeGenerated) == 0 // Sunday and hourofday(TimeGenerated) between (2 .. 6) )
Contextual Filtering â–¼

Add conditions that distinguish malicious from benign.

Example:

// PowerShell alert - add context to reduce FPs | where FileName =~ "powershell.exe" | where ProcessCommandLine has_any ("-enc", "bypass") // Add: Only if NOT from expected management tools | where InitiatingProcessFileName !in~ ( "sccm.exe", "intune.exe", "ansible.exe" ) // Add: Only if from user workstation (not server) | where DeviceName startswith "WS-"
Severity Adjustment â–¼

Lower severity for common FPs instead of disabling entirely.

When to Use:

  • Alert has some value but high FP rate
  • Want to keep visibility but reduce priority
  • Compliance requires alert to exist

Lower from High â†’ Medium so it's still visible but doesn't require immediate response.

Exclusion Management

Exclusion Best Practices

  • Be specific: Narrow exclusions (hash > signer > path)
  • Document why: Record reason for each exclusion
  • Review periodically: Are old exclusions still needed?
  • Require approval: Exclusions reviewed before adding
  • Monitor excluded items: Watch for abuse of exclusions

Exclusion Specificity Hierarchy

Specificity Example Risk
Most Specific (Best) Exclude specific file hash Lowest - very targeted
Specific Exclude signed by specific publisher Low - relies on code signing
Moderate Exclude specific file path Medium - attackers can use same path
Broad Exclude all activity from user High - if user compromised, no alerts
Broadest (Worst) Exclude entire subnet or all servers Very High - large blind spot
Exclusion Anti-Patterns
  • Excluding entire folders like C:\Temp (malware lives there)
  • Excluding all activity from privileged accounts
  • Excluding based on process name alone (can be spoofed)
  • Exclusions without documented justification
  • Never reviewing or removing old exclusions

Regular Review Cadence

  • Weekly: Review high-FP rules for tuning
  • Monthly: Review rule effectiveness metrics
  • Quarterly: Review all exclusions, retire unused rules
  • Annually: Full detection coverage review vs MITRE

KQL Calculate FP rate by alert rule
SecurityIncident | where TimeGenerated > ago(30d) | where Status == "Closed" | summarize TotalAlerts = count(), FalsePositives = countif(Classification == "FalsePositive"), TruePositives = countif(Classification == "TruePositive") by AlertRuleName = tostring(AdditionalData.alertProductNames) | extend FPRate = round(todouble(FalsePositives) / TotalAlerts * 100, 1) | order by FPRate desc
Key Takeaways
  • Tuning is continuous, not one-time
  • Analyze FP patterns before adding exclusions
  • Use most specific exclusion possible (hash > path > user)
  • Document every exclusion with justification
  • Review and prune exclusions quarterly
  • Track FP rate per rule to identify tuning needs
  • Lower severity as alternative to disabling rules
