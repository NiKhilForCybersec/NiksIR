Splunk Alerts & Correlation Searches

Configure effective security alerts, manage notable events, implement risk-based alerting, and reduce alert fatigue in your SOC.

Alert Types

â±ï¸ Scheduled Alerts

Run at defined intervals (every 5 min, hourly, daily)

  • Best for: Most security use cases
  • Performance: Low resource usage
  • Latency: Depends on schedule

ğŸ”´ Real-Time Alerts

Continuous monitoring with immediate notification

  • Best for: Critical, time-sensitive events
  • Performance: High resource usage
  • Latency: Near-instant

ğŸ“Š Rolling Window

Real-time search over a sliding time window

  • Best for: Threshold-based detection
  • Performance: Moderate resource usage
  • Latency: Within window size
Real-Time Alert Considerations

Use real-time alerts sparingly. They consume significant resources and can impact Splunk performance. For most SOC use cases, scheduled alerts running every 5-15 minutes provide adequate coverage with better efficiency.

Creating Effective Alerts

Alert Workflow

  1. Define the search: Start with a tested SPL query
  2. Set trigger conditions: When should this alert?
  3. Configure actions: What happens when it triggers?
  4. Set throttling: Prevent alert storms
  5. Test and tune: Validate and reduce false positives

Trigger Conditions

Condition Use Case Example
Number of results > 0 Any match detection Malware detected, known bad IP
Number of results > N Threshold alerting More than 10 failed logins
Number of results drops below N Health monitoring Log source went silent
Custom condition Complex logic Statistical anomaly

Correlation Search Examples

Brute Force Detection Critical 
index=wineventlog EventCode=4625
| bin _time span=10m
| stats count as failures, 
        dc(Account_Name) as unique_accounts,
        values(Account_Name) as targeted_accounts
  by src_ip, _time
| where failures > 20
| eval description="Brute force attempt: ".failures." failed logins from ".src_ip

Trigger: Results > 0 | Schedule: Every 5 minutes

Successful Login After Failures High 
index=wineventlog (EventCode=4624 OR EventCode=4625)
| stats count(eval(EventCode=4625)) as failures,
        count(eval(EventCode=4624)) as successes,
        latest(_time) as last_event
  by Account_Name, src_ip
| where failures > 5 AND successes > 0
| eval risk_score=failures * 2
| eval description="Potential compromised account: ".Account_Name

Trigger: Results > 0 | Schedule: Every 15 minutes

Suspicious PowerShell Execution Critical 
index=wineventlog EventCode=4688 
  (process_name="powershell.exe" OR process_name="pwsh.exe")
| search CommandLine="*-enc*" OR CommandLine="*bypass*" 
         OR CommandLine="*hidden*" OR CommandLine="*downloadstring*"
         OR CommandLine="*invoke-expression*" OR CommandLine="*iex*"
| eval risk_score=case(
    like(CommandLine, "%downloadstring%"), 80,
    like(CommandLine, "%-enc%"), 70,
    like(CommandLine, "%bypass%"), 60,
    1=1, 50)
| table _time, host, user, process_name, CommandLine, risk_score

Trigger: Results > 0 | Schedule: Every 5 minutes

Large Data Upload High 
index=proxy http_method=POST
| stats sum(bytes_out) as total_uploaded by src_ip, user, dest
| where total_uploaded > 100000000
| eval MB_uploaded=round(total_uploaded/1024/1024, 2)
| eval description="Large upload detected: ".MB_uploaded."MB to ".dest

Trigger: Results > 0 | Schedule: Every 15 minutes

New Admin Account Created Critical 
index=wineventlog EventCode=4720 
| join type=inner Account_Name 
    [search index=wineventlog EventCode=4728 
     | search TargetUserName="Domain Admins" OR 
              TargetUserName="Administrators" OR
              TargetUserName="Enterprise Admins"
     | rename MemberName as Account_Name]
| table _time, host, SubjectUserName, Account_Name, TargetUserName
| eval description="New admin account created: ".Account_Name." by ".SubjectUserName

Trigger: Results > 0 | Schedule: Every 5 minutes

Log Source Health Informational 
| tstats count where index=* by index, sourcetype, _time span=1h
| stats latest(_time) as last_event, count as event_count by index, sourcetype
| eval hours_since_event=round((now() - last_event)/3600, 1)
| where hours_since_event > 4
| eval description="No events from ".sourcetype." in ".hours_since_event." hours"

Trigger: Results > 0 | Schedule: Every hour

Risk-Based Alerting (RBA)

RBA aggregates multiple low-fidelity detections into high-fidelity alerts, reducing alert fatigue while maintaining coverage.

How RBA Works

  1. Risk Rules: Detections assign risk points to entities instead of creating alerts
  2. Risk Accumulation: Points aggregate in the risk index by user/host
  3. Threshold Alert: Notable event fires when entity exceeds threshold
  4. Full Context: All contributing events available for investigation
Risk Rule Template 
# Instead of triggering alert directly, add to risk index
index=wineventlog EventCode=4625
| stats count as failures by Account_Name, src_ip
| where failures > 3
| eval risk_score=failures * 5
| eval risk_object=Account_Name
| eval risk_object_type="user"
| eval risk_message="Multiple failed logins (".failures.") from ".src_ip
| collect index=risk
Risk Threshold Correlation Search 
# Alert when accumulated risk exceeds threshold
| from datamodel:"Risk"."All_Risk"
| stats sum(risk_score) as total_risk,
        values(risk_message) as contributing_events,
        dc(source) as detection_count
  by risk_object, risk_object_type
| where total_risk > 100
| eval urgency=case(
    total_risk > 200, "critical",
    total_risk > 150, "high",
    total_risk > 100, "medium")

  • Reduces alert volume by 80-90%
  • Higher fidelity notable events
  • Full attack chain visibility
  • Contextual investigation

  • Low confidence: 5-20 points

Alert Throttling

Prevent alert storms while ensuring coverage:

Setting Description Example
Suppress results Don't alert on same condition Suppress for 1 hour per src_ip
Suppress fields Group suppression by field values src_ip, user, dest
Suppress period How long to suppress 1h, 4h, 24h
Throttling Best Practices
  • Throttle by the most specific field (user+src combination)
  • Match throttle period to attack patterns (brute force: 1h, recon: 24h)
  • Don't over-throttle critical alerts
  • Review suppressed alerts periodically

Alert Tuning Process

ğŸ“‰ Reducing False Positives

  1. Review closed/FP notables for patterns
  2. Add exclusions (known tools, IPs, users)
  3. Increase thresholds if appropriate
  4. Add time-based filters (business hours)
  5. Refine search logic

ğŸ“ˆ Improving Detection

  1. Review missed incidents
  2. Lower thresholds if missing attacks
  3. Add variant patterns
  4. Expand data sources
  5. Implement RBA for low-signal detections
Exclusion List Pattern
# Use lookup for exclusions
index=wineventlog EventCode=4625
| lookup auth_exclusions src_ip OUTPUT is_excluded
| where isnull(is_excluded) OR is_excluded!="true"
| stats count by Account_Name, src_ip
| where count > 10
