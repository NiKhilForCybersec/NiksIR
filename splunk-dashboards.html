Building Custom Security Dashboards

SOC Overview Dashboard Example

Key panels to include in a SOC monitoring dashboard:

Panel 1: Active Threats (Single Value)
| `notable`
| search status IN (new, "in progress") 
         urgency IN (critical, high)
| stats count as ActiveThreats
Panel 2: Notable Events by Urgency (Pie Chart)
| `notable`
| search status!=closed
| stats count by urgency
| sort -count
Panel 3: Top Triggering Rules (Bar Chart)
| `notable`
| stats count by rule_name
| sort -count
| head 10
Panel 4: Event Trend (Timechart)
index=* sourcetype=*
| timechart span=1h count by index
| head 24
Panel 5: Failed Logins by Source (Table)
index=wineventlog EventCode=4625
| stats count as Failures, 
        dc(Account_Name) as UniqueAccounts 
  by src_ip
| sort -Failures
| head 15

Dashboard Inputs & Tokens

Make dashboards interactive with form inputs that filter all panels:

Simple XML - Time Range Picker
<input type="time" token="time_range">
  <label>Time Range</label>
  <default>
    <earliest>-24h</earliest>
    <latest>now</latest>
  </default>
</input>
Simple XML - Dropdown Filter
<input type="dropdown" token="urgency_filter">
  <label>Urgency</label>
  <choice value="*">All</choice>
  <choice value="critical">Critical</choice>
  <choice value="high">High</choice>
  <choice value="medium">Medium</choice>
  <choice value="low">Low</choice>
  <default>*</default>
</input>

<!-- Use in search -->
<query>
  | `notable` 
  | search urgency=$urgency_filter$
  | stats count by rule_name
</query>
Simple XML - Text Input
<input type="text" token="user_search">
  <label>Username</label>
  <default>*</default>
  <prefix>user=</prefix>
</input>

<!-- Use in search -->
<query>
  index=wineventlog EventCode=4624 $user_search$
  | stats count by host
</query>
Simple XML - Dynamic Dropdown from Search
<input type="dropdown" token="index_select">
  <label>Index</label>
  <search>
    <query>| eventcount summarize=false index=* | dedup index | fields index</query>
    <earliest>-24h</earliest>
    <latest>now</latest>
  </search>
  <fieldForLabel>index</fieldForLabel>
  <fieldForValue>index</fieldForValue>
  <choice value="*">All Indexes</choice>
</input>

Drilldown Configuration

Enable click-through investigation from dashboard panels:

Drilldown to Search
<drilldown>
  <link target="_blank">
    /app/search/search?q=index=wineventlog user=$click.value$
    &earliest=$earliest$&latest=$latest$
  </link>
</drilldown>
Drilldown to Another Dashboard
<drilldown>
  <link target="_blank">
    /app/SplunkEnterpriseSecuritySuite/identity_investigator?
    form.identity=$row.user$
  </link>
</drilldown>
Set Token on Click
<drilldown>
  <set token="selected_user">$click.value$</set>
  <set token="show_details">true</set>
</drilldown>

Visualization Best Practices

Visualization Best For Example Use Case
Single Value Key metrics, counts Active alerts, total events
Timechart Trends over time Event volume, login patterns
Bar Chart Comparisons, rankings Top sources, rule triggers
Pie Chart Distribution/proportions Events by severity, action breakdown
Table Detailed data, drill-down Event lists, investigation queues
Choropleth Map Geographic data Attack origins, user locations

Dashboard Design Tips
  • Put critical metrics at the top left (highest visibility)
  • Use consistent color coding (red=critical, yellow=warning, green=normal)
  • Limit to 6-8 panels per dashboard for focus
  • Include time range selector for flexibility
  • Add drilldowns for investigation workflows

Real-Time Dashboards

Monitor events as they happen with real-time searches:

Real-Time Search Options
# 1-minute sliding window (updated every second)
earliest=rt-1m latest=rt

# 5-minute sliding window
earliest=rt-5m latest=rt

# Real-time all time (resource intensive!)
earliest=rt latest=rt
Real-Time Performance

Real-time searches consume significant resources. Use sparingly and prefer scheduled searches with short intervals (e.g., every 1-5 minutes) for most monitoring needs.

Dashboard Performance Optimization

  • Limit time ranges by default
  • Use report acceleration for complex searches

  • Use fields command to limit data

</search>
