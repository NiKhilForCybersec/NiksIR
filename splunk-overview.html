Splunk Enterprise Security (ES)

ES is the premium security solution that transforms Splunk into a full-featured SIEM. Key features for SOC analysts:

Notable Events

Notable events are the primary unit of work in ES - similar to alerts in other SIEMs:

Field Description Usage
rule_name Correlation search that generated the notable Identify detection logic
urgency Critical, High, Medium, Low, Informational Prioritize response
status New, In Progress, Pending, Resolved, Closed Track investigation state
owner Analyst assigned to the notable Workload distribution
src / dest Source and destination from the event Identify affected systems
risk_score Cumulative risk points for the entity Assess overall threat level
Query Notable Events
| `notable`
| search status="new" urgency IN ("critical", "high")
| table _time, rule_name, src, dest, user, urgency, status
| sort -_time

Risk-Based Alerting (RBA)

RBA aggregates risk scores from multiple detections to surface high-fidelity alerts:

How RBA Works

  1. Risk Rules: Detections assign risk points to entities (users, hosts)
  2. Risk Aggregation: Points accumulate in the risk index
  3. Threshold Alerts: Notable events fire when thresholds exceeded
  4. Context: All contributing events available for investigation
View Risk Scores by User
| from datamodel:"Risk"."All_Risk"
| stats sum(risk_score) as total_risk, 
        values(source) as contributing_rules,
        count as event_count
  by risk_object
| where total_risk > 50
| sort -total_risk

Threat Intelligence Framework

ES integrates threat intelligence for automatic IOC matching:

  • Threat Lists: IP addresses, domains, file hashes, URLs
  • STIX/TAXII: Automated threat feed ingestion
  • Lookups: Custom threat intelligence files
  • Threat Artifacts: Matched IOCs across your data
Search for IOC Matches
| tstats count from datamodel=Network_Traffic 
  where All_Traffic.dest_ip IN 
    [| inputlookup threat_ip_list | fields ip | rename ip as All_Traffic.dest_ip]
  by All_Traffic.src_ip, All_Traffic.dest_ip
| rename All_Traffic.* as *

index=firewall action=blocked

Common Information Model (CIM)

The CIM provides standardized field names across different data sources, enabling consistent searches:

Data Model Key Fields Common Sources
Authentication user, src, dest, action, app Windows Security, AD, VPN, SSO
Network_Traffic src_ip, dest_ip, src_port, dest_port, bytes Firewalls, NetFlow, Zeek
Endpoint dest, process, process_path, user, parent_process EDR, Sysmon, Windows Events
Web url, http_method, status, src, dest Proxy logs, WAF, Web servers
Email src_user, recipient, subject, file_name Exchange, O365, Email gateway
Malware file_hash, file_name, signature, action AV, EDR, Sandbox
Using Data Models with tstats
# Fast search using accelerated data models
| tstats count from datamodel=Authentication 
  where Authentication.action=failure 
  by Authentication.user, Authentication.src
| rename Authentication.* as *
| where count > 10
Performance Tip

Always use tstats with accelerated data models for better performance on large datasets. Standard searches should use indexed fields when possible.

Index Management for Analysts

Understanding indexes helps you search the right data efficiently:

Common Security Indexes
# List all indexes you have access to
| eventcount summarize=false index=* 
| dedup index 
| table index

# Common security index naming conventions
index=main              # Default (avoid in production)
index=wineventlog       # Windows Event Logs
index=firewall          # Firewall logs
index=proxy             # Web proxy logs  
index=dns               # DNS query logs
index=edr               # Endpoint Detection & Response
index=syslog            # Linux/Unix syslog
index=o365              # Microsoft 365 logs
index=aws               # AWS CloudTrail/VPC Flow
index=azure             # Azure Activity/Sign-in logs
Check Index Health
# Check data ingestion rate
| tstats count where index=* by index, _time span=1h
| timechart span=1h sum(count) by index

# Find gaps in data
| tstats count where index=wineventlog by _time span=1h
| timechart span=1h count
| where count < 1000  # Adjust threshold as needed
