Hunting Fundamentals

Threat hunting is the proactive search for threats that have evaded existing detection capabilities. Unlike reactive alert investigation, hunting assumes the adversary is already in your environment.

Reactive SOC

  • Waits for alerts to fire
  • Investigates known-bad indicators
  • Relies on existing detections
  • Catches what tools detect

Proactive Hunting

  • Assumes compromise
  • Searches for unknown threats
  • Develops new detections
  • Finds what tools miss
The Hunting Mindset

"Assume breach" - Start with the premise that adversaries are already in your environment, then look for evidence to prove or disprove that assumption. Absence of evidence is not evidence of absence.

Hunt Maturity Model

Level Description Characteristics
HM0 - Initial No hunting capability Relies entirely on automated alerts
HM1 - Minimal IOC searching Searches for known indicators from threat intel
HM2 - Procedural Following procedures Uses documented hunt playbooks
HM3 - Innovative Creating new hunts Develops original hypotheses, creates detections
HM4 - Leading Automating discoveries Converts hunts to automated detections at scale

Hypothesis Development

A good hunt starts with a good hypothesis. Hypotheses should be specific, testable, and based on threat intelligence or environmental knowledge.

Hypothesis Sources

Intelligence-Driven â–¼

Based on external threat intelligence:

  • APT group TTPs targeting your industry
  • New malware family techniques
  • Vulnerability exploitation in the wild
  • Published threat reports

Example Hypothesis:

"APT29 has been targeting healthcare with spearphishing. We should look for their known C2 domains and PowerShell obfuscation patterns."

TTP-Driven (MITRE ATT&CK) â–¼

Based on adversary techniques:

  • Pick a technique from ATT&CK matrix
  • Research how it manifests in logs
  • Hunt for that behavior pattern

Example Hypothesis:

"Attackers commonly use T1053.005 (Scheduled Tasks) for persistence. We should look for scheduled tasks created via command line or with suspicious actions."

Anomaly-Driven â–¼

Based on deviations from baseline:

  • Statistical outliers in normal behavior
  • First-time events (new process, new connection)
  • Rare patterns in high-frequency data

Example Hypothesis:

"Most lateral movement uses common protocols. We should look for authentication to systems a user has never accessed before."

Environment-Driven â–¼

Based on your specific environment:

  • Known gaps in detection coverage
  • High-value assets to protect
  • Recent changes or vulnerabilities

Example Hypothesis:

"We just deployed new VPN concentrators. We should hunt for unusual VPN access patterns that might indicate compromised credentials."

Hypothesis Quality Checklist

  • Specific: Clearly defines what we're looking for
  • Testable: Can be proved or disproved with available data
  • Relevant: Addresses a realistic threat to our environment
  • Scoped: Has a defined time range and data set
  • Actionable: Results can lead to response or detection improvement

Technique Categories

Stack Counting (Least Frequency Analysis)

Stack counting identifies rare events that may indicate malicious activity. The theory: attackers create artifacts that are uncommon in your environment.

KQL Stack count: Rare processes by execution count
DeviceProcessEvents | where TimeGenerated > ago(7d) | summarize ExecutionCount = count(), DeviceCount = dcount(DeviceName), Users = make_set(AccountName, 5), Devices = make_set(DeviceName, 5) by FileName, FolderPath | where ExecutionCount < 5 // Rare | where DeviceCount == 1 // Only one device | order by ExecutionCount asc
KQL Stack count: Rare network destinations
DeviceNetworkEvents | where TimeGenerated > ago(7d) | where RemoteIPType == "Public" | summarize ConnectionCount = count(), DeviceCount = dcount(DeviceName), Devices = make_set(DeviceName, 5), Processes = make_set(InitiatingProcessFileName, 5) by RemoteIP | where DeviceCount == 1 // Only one device talks to this IP | where ConnectionCount > 10 // But talks frequently | order by ConnectionCount desc

Baselining and Anomaly Detection

KQL Hunt: First-time service installations
// Services seen in last 30 days let KnownServices = DeviceEvents | where TimeGenerated between (ago(30d) .. ago(1d)) | where ActionType == "ServiceInstalled" | extend ServiceName = tostring(AdditionalFields.ServiceName) | distinct ServiceName; // New services in last 24 hours DeviceEvents | where TimeGenerated > ago(1d) | where ActionType == "ServiceInstalled" | extend ServiceName = tostring(AdditionalFields.ServiceName) | extend ServicePath = tostring(AdditionalFields.ServicePath) | join kind=leftanti KnownServices on ServiceName | project TimeGenerated, DeviceName, ServiceName, ServicePath, InitiatingProcessAccountName

Example Hunts

Hunt: Persistence via Registry Run Keys â–¼

Hypothesis:

Attackers establish persistence by adding entries to registry Run keys that execute on login.

MITRE ATT&CK:

T1547.001 - Boot or Logon Autostart Execution: Registry Run Keys

Hunt Query:

DeviceRegistryEvents | where TimeGenerated > ago(7d) | where ActionType == "RegistryValueSet" | where RegistryKey has_any ( @"\CurrentVersion\Run", @"\CurrentVersion\RunOnce" ) | where RegistryValueData has_any ( ".exe", ".dll", ".bat", ".ps1", ".vbs", "powershell", "cmd", "wscript", "mshta" ) // Exclude known software | where RegistryValueData !has "Microsoft" | where RegistryValueData !has "Google" | project TimeGenerated, DeviceName, RegistryKey, RegistryValueName, RegistryValueData, InitiatingProcessFileName, InitiatingProcessCommandLine

Analysis:

  • Review paths in RegistryValueData - are they in unusual locations?
  • Check InitiatingProcess - was it cmd/powershell?
  • Validate executables against known-good lists

Expected False Positives:

  • Legitimate software installations
  • IT deployment tools
  • Startup optimization utilities
Hunt: LSASS Access for Credential Dumping â–¼

Hypothesis:

Attackers access LSASS memory to dump credentials. This is detectable through process access events.

MITRE ATT&CK:

T1003.001 - OS Credential Dumping: LSASS Memory

Hunt Query:

DeviceEvents | where TimeGenerated > ago(7d) | where ActionType == "ProcessAccessed" | where FileName =~ "lsass.exe" // Exclude legitimate security software | where InitiatingProcessFileName !in~ ( "MsMpEng.exe", "csrss.exe", "svchost.exe", "wininit.exe", "wmiprvse.exe" ) | project TimeGenerated, DeviceName, InitiatingProcessFileName, InitiatingProcessCommandLine, InitiatingProcessAccountName | summarize Count = count(), Devices = make_set(DeviceName) by InitiatingProcessFileName | order by Count asc

Suspicious Findings:

  • procdump.exe, taskmgr.exe (with dump), comsvcs.dll
  • Any unsigned process accessing LSASS
  • PowerShell or cmd accessing LSASS
Hunt: C2 Beaconing Patterns â–¼

Hypothesis:

Command and control malware beacons at regular intervals. Detecting regular timing patterns can identify C2.

MITRE ATT&CK:

T1071 - Application Layer Protocol (C2 Communication)

Hunt Query:

DeviceNetworkEvents | where TimeGenerated > ago(24h) | where RemoteIPType == "Public" | summarize Connections = make_list(TimeGenerated, 1000) by DeviceName, RemoteIP, InitiatingProcessFileName | where array_length(Connections) > 20 | mv-apply Connections to typeof(datetime) on ( sort by Connections asc | extend NextConn = next(Connections) | extend IntervalSec = datetime_diff('second', NextConn, Connections) | where isnotnull(IntervalSec) | summarize AvgInterval = avg(IntervalSec), StdDev = stdev(IntervalSec), Count = count() ) | extend CoeffVariation = StdDev / AvgInterval | where CoeffVariation < 0.3 // Low variation = regular | where AvgInterval between (30 .. 3600) // 30s to 1hr | project DeviceName, RemoteIP, InitiatingProcessFileName, AvgIntervalSec = round(AvgInterval, 0), CoeffVariation = round(CoeffVariation, 3), Count | order by CoeffVariation asc

Investigation:

  • Lower coefficient of variation = more regular beaconing
  • Check reputation of RemoteIP
  • Investigate the process making connections
Hunt: Living off the Land (LOLBins) â–¼

Hypothesis:

Attackers use legitimate Windows binaries to download payloads and execute malicious code to avoid detection.

MITRE ATT&CK:

T1218 - System Binary Proxy Execution

Hunt Query:

let LOLBins = dynamic([ "certutil.exe", "mshta.exe", "regsvr32.exe", "rundll32.exe", "msiexec.exe", "wmic.exe", "cscript.exe", "wscript.exe", "bitsadmin.exe", "msbuild.exe", "installutil.exe" ]); let SuspiciousArgs = dynamic([ "http", "ftp", "/download", "-urlcache", "javascript:", "vbscript:", "/i:http", "scrobj.dll", "-decode", "-encode" ]); DeviceProcessEvents | where TimeGenerated > ago(7d) | where FileName in~ (LOLBins) | where ProcessCommandLine has_any (SuspiciousArgs) | project TimeGenerated, DeviceName, FileName, ProcessCommandLine, AccountName, InitiatingProcessFileName | summarize Count = count(), Devices = make_set(DeviceName, 10) by FileName, ProcessCommandLine | order by Count desc

Hunt Outputs

Successful hunts produce actionable outputs. Document findings and convert discoveries into improvements.

Output Types

Output Description Action
True Positive Actual malicious activity found Escalate to incident response
Detection Rule Hunt query becomes automated alert Create analytic rule in SIEM
Visibility Gap Required data isn't being collected Improve logging coverage
Configuration Issue Misconfigurations found during hunt Remediate and document
False Positive Tuning Legitimate activity to exclude Update existing detection rules
Hypothesis Disproved No evidence of threat found Document and move to next hunt

Converting Hunts to Detections

Detection Engineering Cycle
  1. Hunt finds suspicious pattern
  2. Validate pattern against known attacks
  3. Tune query to reduce false positives
  4. Create detection rule from hunt query
  5. Document detection logic and response
  6. Monitor and refine over time
Key Takeaways
  • Hunting is proactiveâ€”assume compromise and look for evidence
  • Good hypotheses are specific, testable, and based on intelligence
  • Stack counting finds rare events that may indicate threats
  • Baselining detects deviations from normal behavior
  • Coefficient of variation < 0.3 suggests regular beaconing (C2)
  • Document all huntsâ€”even negative results have value
  • Convert successful hunts into automated detection rules
