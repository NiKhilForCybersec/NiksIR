Zero Trust Principles

Zero Trust is a security model based on "never trust, always verify." Unlike traditional perimeter security, Zero Trust assumes breach and verifies every access request regardless of source.

Core Principles

Never Trust, Always Verify

Every access request must be authenticated and authorized, regardless of network location.

Assume Breach

Design systems expecting attackers are already inside. Minimize blast radius of compromise.

Least Privilege Access

Grant minimum permissions needed. Use just-in-time and just-enough-access.

Verify Explicitly

Make decisions based on all available data: identity, device, location, behavior.

Traditional vs Zero Trust

Aspect Traditional (Perimeter) Zero Trust
Trust Model Trust inside, verify outside Never trust, always verify
Network Access VPN grants full network access Per-application access only
Authentication Once at perimeter Continuous, per-resource
Lateral Movement Easy once inside Severely restricted
Device Trust Corporate network = trusted Device health verified

Zero Trust Pillars

Identity ‚ñº

Strong identity is the foundation of Zero Trust.

Key Controls:

  • Multi-factor authentication (phishing-resistant preferred)
  • Single sign-on (SSO)
  • Conditional access policies
  • Identity protection and risk detection
  • Privileged identity management

SOC Relevance:

Monitor sign-in logs, identity protection alerts, MFA challenges, and impossible travel detections.

Devices ‚ñº

Verify device health and compliance before granting access.

Key Controls:

  • Device registration and management (MDM/MAM)
  • Compliance policies (encryption, patching)
  • Health attestation
  • EDR/XDR deployment

SOC Relevance:

Monitor device compliance status, EDR alerts, and access attempts from non-compliant devices.

Network ‚ñº

Network segmentation and encrypted communications.

Key Controls:

  • Micro-segmentation
  • Software-defined perimeter
  • Encrypted traffic (TLS everywhere)
  • Network access control

SOC Relevance:

Monitor for lateral movement attempts, segment violations, and unusual network patterns.

Applications ‚ñº

Secure access to applications without exposing the network.

Key Controls:

  • Application proxy / ZTNA
  • Per-app authorization
  • Session controls
  • Cloud app security (CASB)

SOC Relevance:

Monitor application access, unusual usage patterns, and data exfiltration indicators.

Data ‚ñº

Protect data regardless of where it resides.

Key Controls:

  • Data classification and labeling
  • Data loss prevention (DLP)
  • Encryption at rest and in transit
  • Rights management

SOC Relevance:

Monitor DLP alerts, sensitive data access, and potential exfiltration.

SOC Monitoring in Zero Trust

What Changes for SOC

New Visibility

  • Every access request logged
  • Continuous authentication events
  • Device compliance status
  • Per-app access patterns
  • Risk scores for users/devices

New Challenges

  • Higher volume of auth events
  • More complex log sources
  • Policy-based detections needed
  • Baseline per-user/per-app
  • False positives from legitimate blocks

Key Alerts in Zero Trust

Alert Indicates Response
Conditional Access block Policy prevented risky access Review if legitimate, investigate if unusual
Device compliance failure Device doesn't meet security requirements May indicate compromise or misconfiguration
Risky sign-in detected Identity protection triggered Investigate, may require password reset
MFA bypass attempt Attacker trying to avoid MFA High priority investigation
Unusual application access User accessing apps they don't normally use Verify with user or manager

Investigation Approach

KQL Conditional Access blocks investigation
SigninLogs | where TimeGenerated > ago(24h) | where ConditionalAccessStatus == "failure" | summarize BlockCount = count(), Apps = make_set(AppDisplayName), Locations = make_set(Location) by UserPrincipalName, ResultType, IPAddress | where BlockCount > 5 | order by BlockCount desc
KQL Access from non-compliant devices
SigninLogs | where TimeGenerated > ago(7d) | where DeviceDetail.isCompliant == false | where ResultType == "0" // Successful despite non-compliance | project TimeGenerated, UserPrincipalName, AppDisplayName, DeviceDetail, IPAddress, ConditionalAccessPolicies | order by TimeGenerated desc

  • Every access decision based on identity + device + location + behavior
  • Assume breach‚Äîdesign to minimize blast radius
